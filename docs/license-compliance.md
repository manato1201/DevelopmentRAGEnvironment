**ライセンス: MIT（原作者: karaage, 2025）**

```
MIT License

```

[karaage0703/mcp-rag-server](https://github.com/karaage0703/mcp-rag-server) の検索エンジンコードをこのリポジトリに直接取り込み（ベンダー統合）、`scripts/document_processor.py` / `scripts/embedding_generator.py` / `scripts/rag_service.py` / `scripts/mcp_server.py` などとして同梱しています（2026-06-30、外部リポジトリへの依存を解消）。MIT ライセンスは改造・再配布・組み込みを許可しているため、ソースコードを直接コピーして本リポジトリの一部にすることに問題はありません。**「外部ツールへの依存」ではなく「自社コードに組み込んだサードパーティ由来のコード」という扱いになります。**

### 許可されること
- 商用製品への組み込み・販売
- ソースコードの改造・カスタマイズ
- 再配布・サブライセンス

### 義務
- **配布時にのみ**: 取り込んだコードの著作権表示（上記 MIT License テキスト）を `LICENSE` または `NOTICE` ファイル等に同梱すること
- 内部ツールとして使うだけなら配布不要 → 義務なし
- ベンダー統合（コードをコピーして改変）しても、MIT ライセンスである限りソース開示義務は発生しない

### 依存パッケージのライセンス

ベンダー統合コードが利用するパッケージ（`pyproject.toml` で直接宣言）はすべて許容的ライセンス:

## 3. PySide6（Houdini パネルで使用）

**ライセンス: LGPL v3（または商用ライセンス）**

### 条件
- **動的リンク（通常の import）**: 自社コードのソース開示不要。Houdini の Python パネルは import 形式なので**これに該当**
- **PySide6 本体を改変した場合**: その差分のみ LGPL 準拠で公開が必要
- **外部配布しない社内ツール**: 開示義務なし

### ライセンス表示義務
外部にソフトウェアを配布する場合は、製品のドキュメントやライセンス画面に以下を記載:
```
This software uses PySide6, licensed under LGPL v3.
https://www.qt.io/licensing/
```

### 商用ライセンスへの切り替え
Qt 商用ライセンス（有料）に切り替えると LGPL 義務から完全に解放される。大規模商用製品で義務回避を重視する場合は検討。

## 6. Claude API（Anthropic）

**利用規約ベース。従量課金。**

### 商用利用
- API 経由での商用利用は許可されている
- 利用規約の禁止事項（有害コンテンツ生成等）を遵守する必要あり

### データ取り扱い（重要）
- **API 経由のデータはモデル学習に使用されない**（claude.ai Consumer 製品とは異なる）
- エンタープライズ向けに DPA（データ処理契約）の締結が可能
- 機密情報・個人情報を扱う場合は DPA 締結を推奨

### 料金・上限
- トークン従量課金（ユーザー数上限なし）
- レート制限あり（Tier に応じて上限が変化）
- 大量利用の場合は Enterprise プランで専用レート制限を設定可能

## 10. コンテンツ動的生成機能への権利リスク波及対策

> 対象: [docs/content-generation.md](content-generation.md) の houdini21（Houdiniチュートリアル自動生成）・BrainTQ（ミニゲーム動的生成）機能。

RAG機能の商用展開に伴うライセンスリスクは、これまでの章（MIT/Apache等の依存パッケージ）とは別の性質を持つリスクとして、コンテンツ動的生成機能に波及しうる。RAG検索結果は生成AIへの入力（コンテキスト）として使われるため、**RAGコーパスの中身そのものの出典**が生成物の権利リスクに直結する。

### 10.1 問題の構造

もし RAG コーパス（houdini21DB等）の中に著作権保護されたテキストの丸写し（例: SideFXの公式ヘルプドキュメントをそのままコピーしたもの）が混ざっていると、生成されたチュートリアルにその文章が逐語的に出力されるリスクがある。これは依存パッケージのライセンス（MIT/Apache等）とは別次元の問題であり、**Notion上のコンテンツの作成過程**が焦点になる。

### 10.2 対策①: RAGコーパスの「生成用途で使える」ホワイトリスト化

| namespace | 生成機能（houdini21/BrainTQ）で使用可否 | 理由 |
|-----------|---------------------------------|------|
| houdini21 | 要出典棚卸し後に許可 | Notionに手入力した内容が「公式ドキュメントの要約・自分の言葉での説明」であることを確認する必要がある |
| tool_docs / research | 原則不可（チャット回答専用に限定） | 外部記事の引用・要約が混在している可能性が高く、生成物への混入リスクが高い |
| game_info / team_notes | 不可 | 社内限定情報。外部提供物に混ぜるべきではない |
| ミニゲームDocumentマニュアル（自社コード仕様のみ） | 許可 | 自社コードの仕様書であり、他者の著作物ではない |

houdini21DBの全ページについて、「SideFXの文章をコピーしていないか」の棚卸しを一度実施することを推奨する。これは実装ではなく、Notionの中身のレビュー作業である。

### 10.3 対策②: 生成パイプラインへの出典逸脱チェックの組み込み

生成されたMarkdown/JSONを保存する直前に、RAGで取得した各チャンクとの n-gram一致率チェック（例: 連続20単語以上の一致がないか）を自動実行し、閾値を超えたら警告してその段落を要約し直させる工程を、保存処理（houdini21では`tutorial_agent.py`）に追加する。

### 10.4 対策③: 生成物の権利帰属の明文化

商用契約・利用規約に以下を明記する方針とする。

- 生成されたチュートリアル・ノードグラフ・ミニゲームコンテンツの著作権は、生成を実行した顧客に帰属する（Anthropic API の利用規約上、API出力の権利は基本的に呼び出し側に帰属するため、この方針と矛盾しない）
- ただしRAGコーパス（houdini21DB等）自体の著作権・利用許諾はサービス提供者側の責任範囲であり、10.2/10.3で担保する
- 生成物にSideFX/Unityのロゴ・サンプルアセット（テクスチャ等）を含めない（Houdiniのサンプルシーンに含まれるテクスチャ等は多くの場合「Houdini内での使用のみ許諾」であり、外部配布不可のケースがある）

### 10.5 ノードタイプ名の扱いについて

生成物には`mountain::2.0`のようなSideFXのノードタイプ名が大量に含まれるが、これは事実の記述であり著作物性がないため問題にならない（API名・関数名の列挙が著作権侵害にならないのと同じ整理）。この点は対策不要。

### 10.6 チェックリスト

| 対応 | 優先度 | 担当 |
|------|--------|------|
| houdini21DB 全ページの出典棚卸し（コピペ文章がないか） | 高・商用化前必須 | ドキュメント作成者本人（内容の是非は第三者には判断できないため） |
| n-gram出典逸脱チェックの実装 | 中 | 実装担当 |
| 生成namespaceホワイトリストの実装（10.2の表をコードに反映） | 中 | 実装担当 |
| 利用規約への権利帰属条項の追記（法的文言） | 高 | 弁護士確認推奨。ドラフトは実装担当が作成可能だが最終文言は要専門家確認 |

## 11. 「ソース開示が必要になる」ケースの整理

商用化で最も怖い「コードを開示しなければならない」状況が発生する条件:

| 条件 | 発生するか |
|------|-----------|
| GPL ライセンスのパッケージを使う | **発生しない**（本システムに GPL なし）|
| AGPL ライセンスのパッケージを使う | **発生しない**（AGPL なし）|
| PySide6 を改変して配布する | **改変部のみ**開示が必要 |
| PySide6 を改変せず使う | 開示不要 |
| MIT/Apache 2.0 のコードを改変する | 開示不要 |

**結論: 現在の構成でソースコードの全面開示が義務になるケースはない。**  
PySide6 を改造しない限り、自社コードは完全にクローズドにできる。
