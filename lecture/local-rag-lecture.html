ローカルRAG環境構築
ChromaDB + Windows Native + 単一リポジトリ完結

Prerequisites

前提条件

必要な環境
  • Windows 11
  • Claude Desktop インストール済み
  • Obsidian インストール済み
  • ディスク空き 5GB 以上
不要なもの（旧構成から削除）
  • WSL2 / Ubuntu
  • Docker Desktop
  • PostgreSQL / pgvector
  • 外部リポジトリ（mcp-rag-server）のクローン
  • Python の別途インストール（uv が管理）

なぜ単一リポジトリ構成に統合したのか

以前は検索エンジン本体を外部リポジトリ mcp-rag-server としてクローンし、ブリッジスクリプトがサブプロセスとして起動・JSON-RPC over stdio で通信していた。現在は検索エンジンのコード（document_processor.py / embedding_generator.py / rag_service.py / mcp_server.py など）をこのリポジトリに直接ベンダリング（MITライセンスのため取り込み可）し、単体で完結する構成にした。クローン・パッチ適用・別ディレクトリでの依存管理が一切不要になった。

Step 01

uv インストール

uv は Python の高速パッケージマネージャー。pip より依存解決が安定しており、Python バージョン管理も一括して行える。

1
winget でインストール PowerShell
PowerShell
PS> winget install --id=astral-sh.uv -e
2
バージョン確認 PowerShell
PowerShell
PS> uv --version
uv 0.x.x  ← バージョンが表示されれば OK
Step 02

依存パッケージの同期

このリポジトリの pyproject.toml に検索エンジンの依存（ChromaDB・sentence-transformers・rank-bm25・sudachipy 等）が直接宣言されている。外部リポジトリのクローンも個別の依存インストールも不要で、リポジトリ直下で uv sync するだけで完結する。

1
依存関係を同期 PowerShell
PowerShell — DevelopmentRAGEnvironment
PS> cd C:\Users\YOUR_USERNAME\Desktop\GameDevelopment\DevelopmentRAGEnvironment
PS> uv sync

Python バージョンの自動管理

pyproject.toml の requires-python が >=3.13 に設定されており、uv が自動で適切なバージョンを用意する。3.13 以降には sentencepiece の pre-built wheel が提供されているため、CMake ビルドエラーは発生しない。初回は sentence-transformers 等のダウンロードで数分かかる。

Step 03

環境変数の設定

このプロジェクトは .env ファイル / load_dotenv() を使用しない方針。設定はすべて OS の環境変数として渡す。デフォルト値のままで問題なければこの手順はスキップしてよい。

1
主な環境変数一覧
変数デフォルト説明
SOURCE_DIRlocalRAG/インデックス化対象ディレクトリ
PROCESSED_DIRlocalRAG/_rag_dashboard/.processed処理済みファイルの保存先
CHROMA_PATHdata/chromaChromaDB データ保存先
EMBEDDING_MODELintfloat/multilingual-e5-large埋め込みモデル名
EMBEDDING_DIM1024埋め込み次元数
EMBEDDING_PREFIX_QUERYquery:検索クエリ用プレフィックス
EMBEDDING_PREFIX_EMBEDDINGpassage:文書埋め込み用プレフィックス
ANTHROPIC_API_KEYなしrag_local_bridge.py 使用時に必須
Step 04

Obsidian vault の設定

Obsidian フォルダ構成 = ChromaDB namespace

vault 内の最上位フォルダ名がそのまま ChromaDB の Collection 名（namespace）になる。_ や . で始まるフォルダは管理用として自動スキップされる。

1
vault フォルダ構成
DevelopmentRAGEnvironment\localRAG\   ← Obsidian で開くフォルダ
├── chat_logs\        ← namespace: chat_logs
├── tutorials\        ← namespace: tutorials
├── personal_notes\   ← namespace: personal_notes
│   ├── progress\
│   └── ideas\
├── private_docs\     ← namespace: private_docs
├── _rag_dashboard\   ← 管理専用（インデックス対象外）
│   ├── index_status.md
│   ├── namespace_map.md
│   └── .processed\
└── _templates\       ← テンプレート（インデックス対象外）
2
Obsidian でフォルダを開く Obsidian

Obsidian を起動 → 「別のボルトを開く」→ フォルダを選択 → localRAG\ を指定。

3
テンプレートプラグインを有効化 Obsidian

設定 → コアプラグイン → テンプレート: オン

設定 → テンプレート → テンプレートフォルダの場所: _templates

YAML フロントマターに以下を含めるとステータス管理が可能：

---
status: active   # active / stale / archived
rag_indexed: false
expires: 2026-12-31  # オプション
---
Step 05

インデックス化

旧構成との違い

pgvector 版では 4 時間以上かかる初回インデックス化が必要だったが、ChromaDB 版は Obsidian vault が空のままインデックス化を実行でき（0件で正常終了）、ノートを追加するたびに差分インデックスすれば良い。

PowerShell — DevelopmentRAGEnvironment
PS> uv run python scripts\rag_cli.py index
インデックス化が完了しました
- ドキュメント数: 1
- 処理時間: 8 秒   ← ファイルを追加後の例

# 差分のみ更新する場合
PS> uv run python scripts\rag_cli.py index --incremental

# ドキュメント数の確認
PS> uv run python scripts\rag_cli.py count

自動スキップされるフォルダ

_rag_dashboard\・_templates\・.processed\ は _namespace_from_path() で自動除外される。手動での除外設定は不要。

Step 06

watchdog（自動インデックス化）の起動

Obsidian でノートを保存するたびに、auto_index.py が変更を検知して自動でインデックスを更新する。

1
バックグラウンドで起動 PowerShell

scripts\auto_index.py はこのリポジトリにすでに含まれているため、コピー作業は不要。リポジトリ直下から直接起動する。

PowerShell — DevelopmentRAGEnvironment
PS> Start-Process -NoNewWindow -FilePath "uv" -ArgumentList "run python scripts/auto_index.py"

python scripts\auto_index.py では起動しないこと

uv の仮想環境外で実行すると ModuleNotFoundError が発生する。必ず uv run 経由で起動すること。

2
動作確認 Obsidian

Obsidian でノートを保存後、_rag_dashboard\index_status.md と namespace_map.md のタイムスタンプが更新されれば watchdog が正常に動作している。

Step 07

Claude Desktop への登録

1
設定ファイルを開く Windows

Everything などで以下を検索して開く：

claude_desktop_config.json

通常のパス：

C:\Users\YOUR_USERNAME\AppData\Roaming\Claude\claude_desktop_config.json
2
設定を書き換える テキストエディタ

YOUR_USERNAME を自分のユーザー名に変更する：

claude_desktop_config.json
{
  "mcpServers": {
    "local-rag-server": {
      "command": "uv",
      "args": [
        "run",
        "--directory",
        "C:\\Users\\YOUR_USERNAME\\Desktop\\GameDevelopment\\DevelopmentRAGEnvironment",
        "python",
        "scripts/rag_mcp_server.py"
      ]
    }
  }
}

旧構成との違い

旧版は外部リポジトリ mcp-rag-server をクローンしたディレクトリを指していたが、新版はこのリポジトリ自身の scripts/rag_mcp_server.py を直接指定する。クローンも別ディレクトリでの依存管理も不要。

3
動作確認 Claude Desktop

Claude Desktop を完全再起動後、設定画面で running と表示されれば完了。

確認プロンプト

local-rag-server でインデックス内のドキュメント数を確認してください。

Troubleshooting

トラブルシューティング

uv sync で sentencepiece のビルドが失敗する

pyproject.toml の requires-python が >=3.13 になっているか確認する。Python 3.13 以降であれば pre-built wheel が提供されているためビルド不要。

'NoneType' object has no attribute 'get_or_create_collection'

vector_database.py の initialize_database() が self.connect() を呼んでいるか確認する（scripts/vector_database.py は変更していなければ問題ない）。

ChromaDB コレクション名エラー（Got: C:\）

SOURCE_DIR 環境変数が正しく設定されていない。

  • SOURCE_DIR が絶対パスで設定されているか確認（デフォルトでは localRAG/ が使われる）
  • 設定後は .processed\ キャッシュを削除して再インデックス
PowerShell
PS> Remove-Item -Recurse -Force localRAG\_rag_dashboard\.processed
PS> uv run python scripts\rag_cli.py index

watchdog が反応しない

PowerShell
# uv run 経由で起動されているか確認
PS> Get-Process python

# 止まっていれば再起動
PS> Start-Process -NoNewWindow -FilePath "uv" -ArgumentList "run python scripts/auto_index.py"

Claude Desktop で MCP が running にならない

  • claude_desktop_config.json のパスが正しいか確認（\\ でエスケープ）
  • PowerShell で uv run --directory "C:\...\DevelopmentRAGEnvironment" python scripts/rag_mcp_server.py が単体で動くか確認
  • Claude Desktop をタスクトレイから完全終了して再起動

インデックス化で _rag_dashboard が処理される

  • _namespace_from_path() が SOURCE_DIR 環境変数を読んで相対パスに変換しているか確認

検索時に KeyError: 'chunk_index' / KeyError: 'document_id' が出る

scripts/vector_database.py の search() / get_adjacent_chunks() / get_document_by_file_path() がこれらのキーをトップレベルで返しているか確認する。

日本語クエリが文字化け / TextEncodeInput エラー（MCP 登録時のみ）

scripts/rag_mcp_server.py の冒頭に UTF-8 パッチ（io.TextIOWrapper）が含まれているか確認する。Windows のデフォルトエンコーディングは CP932 のため、これがないと stdin/stdout で日本語が文字化けする。

エラー例
received query: '繝輔か繝ｼ繝...'   ← 文字化けしている
TextEncodeInput must be Union[TextInputSequence, ...]

インデックス化後も検索結果が増えない（HNSW キャッシュ）

ChromaDB の HNSW インデックスはメモリにロードされる。別プロセスが upsert しても起動中のサーバー側には反映されない。

インデックス化後は必ず MCP サーバー / ブリッジを再起動する。
Claude Desktop: タスクトレイから完全終了 → 再起動
rag_local_bridge.py: プロセスを再起動
