Step 02

依存パッケージの同期

このリポジトリの pyproject.toml に検索エンジンの依存（ChromaDB・sentence-transformers・rank-bm25・sudachipy 等）が直接宣言されている。外部リポジトリのクローンも個別の依存インストールも不要で、リポジトリ直下で uv sync するだけで完結する。

1
依存関係を同期 PowerShell
PowerShell — DevelopmentRAGEnvironment
PS> cd C:\Users\YOUR_USERNAME\Desktop\GameDevelopment\DevelopmentRAGEnvironment
PS> uv sync

Python バージョンの自動管理

pyproject.toml の requires-python が >=3.13 に設定されており、uv が自動で適切なバージョンを用意する。3.13 以降には sentencepiece の pre-built wheel が提供されているため、CMake ビルドエラーは発生しない。初回は sentence-transformers 等のダウンロードで数分かかる。

Internals

内部アーキテクチャ

LocalRAGは5つのコンポーネントが連携して動作する。Obsidian Vault内のMarkdownファイルを起点に、ChromaDBにベクトルとして格納し、MCPまたはHTTPブリッジ経由でクライアントから検索できる構成になっている。

📁 → ⚙️ → 🔢 インデックス側
  • document_processor.py: ファイルスキャン・チャンク分割・差分検出（SHA-256）
  • embedding_generator.py: multilingual-e5-large で1024次元ベクトル生成
  • vector_database.py: ChromaDBへの格納・検索ラッパー
🗄️ → 🔌 → 💬 検索側
  • rag_service.py: 検索・インデックス処理の中核（RAGService）
  • rag_mcp_server.py: Claude Desktop 向け MCP stdio サーバー
  • rag_local_bridge.py: Unity/Houdini 向け HTTP ブリッジ（rag_service.py を直接 import）

2026-06-30 の構成変更点

以前は rag_local_bridge.py が外部リポジトリ mcp-rag-server をサブプロセスとして起動し、MCP（JSON-RPC over stdio）で通信していた。現在は LocalRAGClient クラスが rag_service.py の create_rag_service_from_env() を直接 import して呼び出すだけになっており、サブプロセスもMCPハンドシェイクも発生しない。MCP stdio が必要なのは Claude Desktop 登録時（rag_mcp_server.py）のみで、HTTPブリッジの経路では使われない。

インデックスパイプライン

RAGService.index_documents() が中心となり、DocumentProcessor・EmbeddingGenerator・VectorDatabaseを順に呼び出す。

1
ディレクトリスキャン

SOURCE_DIR 配下を再帰的にスキャンする。最上位フォルダ名がNamespace（ChromaDBコレクション名）になる。_ または . で始まるフォルダは除外される。対応形式は .md / .txt / .pptx / .docx / .pdf（.pptx/.docx/.pdfはmarkitdownで先にMarkdown変換）。

2
差分チェック（incremental mode）

各ファイルのSHA-256ハッシュ・mtime・sizeを計算し、前回の記録と比較してスキップ判定する。新規・変更ファイルのみ処理する。

3
チャンク分割

テキストを500文字ずつに分割し、前後100文字のオーバーラップを持たせる。文の途中で切れないよう \n または 。 を探して文境界で調整する。

4
エンベディング生成

全チャンクに "passage: " プレフィックスを付加し、multilingual-e5-largeでベクトル化する。バッチ処理で効率化されている。

5
ChromaDB格納

batch_insert_documents() でチャンクテキスト・ベクトル・メタデータをまとめて書き込む。CHROMA_PATH（デフォルト data/chroma）にファイルとして永続化され、HNSWインデックスはメモリ上で高速検索できる状態になる。

チャンク分割の詳細

Chunk 0 (0〜500字) ──overlap100字── Chunk 1 (400〜900字) ──overlap100字── Chunk 2 (800〜1300字) ──…

隣接チャンク間で100文字を共有することで、チャンク境界をまたぐ文脈が失われない。検索ヒットしたチャンクの前後が自然に繋がる。

1ファイルに複数トピックを混在させるとスコア差が消える

例: 1ファイルに複数の独立したトピックが混在 → 全チャンクが同一ブランド・同一トピックとして均一にエンコードされ、スコアが団子状態になる。トピックごとにファイルを分割すると、チャンクが専門的な意味空間に集中し、スコア差が拡大して精度が向上する。

エンベディングのプレフィックス設計

intfloat/multilingual-e5-large はcontrastive learningで学習されており、「passage:」と「query:」の非対称な関係を学習している。同じプレフィックスで両方を計算すると類似度が低下するため、用途で使い分ける。

Python embedding_generator.py
# ドキュメントをインデックス化するとき
def generate_embeddings(self, texts):
    prefixed = ["passage: " + t for t in texts]
    return model.encode(prefixed, normalize_embeddings=True)

# ユーザーのクエリを検索するとき
def generate_search_embedding(self, query):
    prefixed = "query: " + query
    return model.encode(prefixed, normalize_embeddings=True)

検索パイプライン

RAGService.search() がクエリ受信から結果返却までの全工程を管理する。クエリに query: プレフィックスを付加 → ベクトル化 → ChromaDBでHNSW近傍探索 → （with_context=True なら）前後チャンクをマージして返却する。

オプションデフォルト説明
limit5返す検索結果の最大件数
with_contextFalseヒットしたチャンクの前後チャンクも取得する
context_size1前後に取得するチャンク数
full_documentFalseヒットしたファイルの全チャンクを返す

類似度スコアの見方

ChromaDBはL2距離で検索し、1 - (distance / 2) でコサイン類似度（0〜1）に変換して返す。85%以上が「関連性が高い」の目安。80〜85%はノイズが混入し始めるゾーン。トピック分割によりスコア差が広がり精度が向上する。

Namespaceシステム

ChromaDBのコレクション（検索対象グループ）は、SOURCE_DIR 直下の最上位フォルダ名が自動的に割り当てられる。MCP/HTTPの検索呼び出し時に namespace を指定すると特定コレクションのみを検索でき、省略すると全コレクションが対象になる。

差分インデックス（増分更新）

毎回全ファイルを処理すると時間がかかるため、--incremental を指定すると変更ファイルのみを処理する。対象ファイルのSHA-256ハッシュ・mtime・sizeを計算 → 記録と照合してスキップ判定 → 変更があれば旧チャンクを削除して新チャンクを挿入、という流れ。auto_index.py（watchdog）は常にincrementalモードで動作する。

MCP通信とWindows対応

Claude Desktop と rag_mcp_server.py はModel Context Protocol (MCP) のstdio transportで通信する。Windows標準の文字コードはCP932（Shift-JIS）であり、日本語を含むJSONをstdoutにそのまま書くと文字化けしてMCPが壊れる。rag_mcp_server.py の冒頭にUTF-8強制パッチが当たっている。

Python rag_mcp_server.py — UTF-8パッチ
import sys, io
# stdoutをUTF-8に強制（Windows CP932問題の回避）
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")
sys.stderr = io.TextIOWrapper(sys.stderr.buffer, encoding="utf-8")

HTTPブリッジ経路（8.2）ではMCPは使われない

rag_local_bridge.py は LocalRAGClient 経由で rag_service.py を直接 import して呼ぶため、stdio/JSON-RPCのオーバーヘッドもCP932の問題も発生しない。CP932パッチが必要なのは Claude Desktop の MCP 登録（8.1）経路のみ。

# メインタイトル（frontmatterのtitleと揃える）

ここに本文を書く。

## セクション1
内容...

rag_indexed: true の確認を忘れずに

新規作成時にテンプレートを使わないと rag_indexed: false のままになることがある。インデックス化しても内容が正式登録されていない「ドラフト状態」になるため、作成後は必ず確認する。

ドキュメント設計 — 検索精度を上げるコツ

RAGの検索精度はドキュメントの書き方・分割の粒度に強く依存する。原則は「1ファイル = 1トピック」。

✓ GOOD — トピック分割
  • unity_shader_basics.md
  • unity_shader_urp.md
  • unity_particle_system.md
✗ BAD — 1ファイルに全部
  • unity_all.md（シェーダー・パーティクル・アニメ・UIすべて混在）
  • → 全チャンクが均一にエンコードされ、検索スコアが80〜83%に団子状態になる
文字数（本文）チャンク数の目安評価
〜800文字1〜2チャンク小さすぎ — 文脈が薄く精度が不安定
800〜3,000文字2〜8チャンク理想 — 1トピックを充分に説明できる量
3,000〜6,000文字8〜14チャンクやや大きい — サブトピックで分割検討
6,000文字〜14チャンク〜分割推奨 — 必ずトピック分割する

インデックス実行

ドキュメントが揃ったらリポジトリのルートでインデックスコマンドを実行する。検索エンジンはこのリポジトリに統合済みのため、外部リポジトリへの移動は不要。

PowerShell — DevelopmentRAGEnvironment
# 初回はフルインデックス
PS> uv run python scripts\rag_cli.py index

# 2回目以降は差分インデックスが高速
PS> uv run python scripts\rag_cli.py index --incremental

# チャンク数の確認
PS> uv run python scripts\rag_cli.py count

clear コマンドは全データを削除する

uv run python scripts\rag_cli.py clear を実行するとChromaDB内の全コレクションが削除される。特定Namespaceだけを消す機能は現在なし。clear後は必ずフルインデックスを実行すること。

動作確認 — スコアで精度を判定する

スコアレンジ判定対応
88%〜良好そのまま使える
83〜88%普通許容範囲。さらに上げたければトピック分割を細かく
78〜83%要注意関係ない文書がヒットしている可能性。ファイル粒度を見直す
〜78%NGドキュメント内容か分割粒度に問題あり。設計から見直す

上位5件のスコア差が3%以上あれば絞り込みが効いている。差が1%未満の場合はトピックが混在しているサイン。

Management Design

Obsidian管理 — 可視化・運用設計

Obsidian vault = SOURCE_DIR とすることでファイルツリーがそのままインデックス対象の一覧になる。watchdogがインデックス後にダッシュボードノートを自動更新することで、ChromaDBの状態をObsidianから常に確認できるようにする設計。

解決している課題

何がインデックスされているか分からない・namespace別の状態が把握できない・更新漏れが発生しやすい、といった問題を vault構造 + frontmatter + watchdog書き戻しの3層設計で解決する。

frontmatterによる鮮度管理

各ノートの先頭にYAMLフロントマターを付けて、インデックス状態と有効期限を管理する。

Markdown tutorials/houdini_vex_wrangle.md
---
title: Houdini VEX wrangle 基礎メモ
namespace: tutorials
status: active
created: 2026-06-10
updated: 2026-06-10
expires: 2026-12-10
tags: [houdini, vex, tutorial]
rag_indexed: false
---

## 概要
...
フィールド値説明
statusactiveインデックス化する（デフォルト）
statusstale期限切れ・要確認。警告ログ付きでインデックス
statusarchivedインデックス化をスキップ
expires日付この日付を過ぎるとwatchdogがstaleに自動変更
rag_indexedfalse→trueインデックス完了後にwatchdogが書き換える

expiresの設定目安

チャット履歴: 90日 / チュートリアル: 180日 / 個人メモ: 無期限。古いチュートリアルが検索に混入するのを防ぐのが主目的。

ダッシュボードノート仕様

scripts/auto_index.py がインデックス完了ごとに _rag_dashboard/ 内のファイルを自動更新する。Obsidianで開くだけで現在のRAG状態が分かる。

index_status.md
  • 最終更新日時・総ファイル数
  • namespace別件数テーブル
  • staleファイルの一覧（Obsidianリンク付き）
namespace_map.md
  • namespace別のファイル一覧
  • インデックス済み: ✓ / 未: ○
  • status と tags を表示

運用フロー

1
Obsidianでノートを作成・編集

frontmatterに namespace と expires を記載して保存する。

2
watchdog（auto_index.py）が自動検知

保存と同時に scripts/rag_cli.py 経由の差分インデックス化が走る（外部リポジトリへの依存はない）。

3
frontmatterとダッシュボードが自動更新

rag_indexed: true に書き換わり、index_status.md が最新になる。

月次メンテナンス

  1. _rag_dashboard\index_status.md を開いて stale ファイルを確認
  2. 各ファイルを開いて内容を精査（削除 or 更新）
  3. 更新した場合は updated と expires を書き換えて status: active に戻す
Use Case

Houdini RAG活用

大規模言語モデルは「知識の鮮度」と「専門ドメインの正確性」という根本的な課題を抱える。Houdiniの場合、新しいノードを知らない・存在しないパラメータをハルシネーションする・バージョン間の情報混在といった問題が起きやすい。RAGはこれらを「検索して、それを根拠に生成する」アプローチで抑制する。

ゲーム開発での実用価値

Houdini MCP でノードを自動配置する際のボトルネックは「存在しないノード・パラメータを指定してしまうこと」。Houdiniヘルプドキュメントをこのリポジトリの houdini21 Namespace としてインデックス化しておくことで、LLMは正確なノード名とパラメータ名を参照しながらHoudiniを操作できる。

MCP設計思想 — 検索タイミングをLLMに委ねる

MCPの重要な設計原則は「LLMはいつ検索すべきかを自分で判断する」という点。RAGの検索タイミングをLLMの推論に委ねることで、「まず何が必要かを考え、必要なノードをリストアップしてからヘルプを検索する」という2段階推論が自然に実現される。これは従来のパイプライン型RAG（検索→プロンプト構築→LLM呼び出しを1本道で固定実装する方式）にはない柔軟性。

Claude Desktop への MCP 登録（現行構成）

このリポジトリ単体で完結する現行構成では、外部リポジトリのクローンやWSL内サーバー起動は不要。「Claude Desktop への登録」の手順どおり scripts/rag_mcp_server.py を指定するだけでよい。

使い方のコツ

問題対策
抽象的タスクで検索が失敗する「必要なノードを列挙してから各ヘルプを検索して」と段階を踏ませる
回答がチャット欄に流れて追いにくい「Markdownファイルに出力して」と明示する
検索スコアが低い・関係ない結果が混ざるクエリに具体的なノード名・パラメータ名を含める（固有名詞は省略しない）

確認問題

理解度チェック

1. multilingual-e5-largeで検索クエリに付与すべきプレフィックスは何か。
2. コサイン類似度が0.0であることは2つのテキストがどのような関係にあることを示すか。
3. HNSWアルゴリズムがブルートフォース検索より優れている点を計算量の観点で説明せよ。
4. MCP環境でRAGを構成する利点を、従来のパイプライン型RAGと比較して説明せよ。
5. このリポジトリでHTTPブリッジ経路がMCP/JSON-RPCを使わなくなった理由を説明せよ。

UI Implementation

UI実装（Unity / Houdini）

LocalRAGへのアクセスはHTTPブリッジ（rag_local_bridge.py、localhost:8766）経由で行う。Unity・Houdiniのどちらも同じHTTP APIを叩くシンプルな構造で、外部ライブラリへの依存を最小限にしている。

rag_local_bridge.py のエンドポイント設計

メソッドパス認証用途
GET/health不要生存確認・総チャンク数取得
POST/query必須（X-API-Key）検索クエリ実行
GET/graph必須namespace別の関係グラフをJSONで取得
GET/api/score必須ユーザーのトピック別理解度スコア取得
POST/api/score必須理解度スコア更新（success true/falseで増減）

LocalRAGClient は in-process 直接呼び出し

rag_local_bridge.py 内の LocalRAGClient クラスは rag_service.py の create_rag_service_from_env() を直接 import して呼び出す。外部プロセスの起動やMCP通信は発生しない。HTTPリクエストを受けたハンドラがこのクライアントのメソッドを直接呼ぶだけのシンプルな構造。

Unity エディタ拡張

C# の EditorWindow + HttpClient（async/await）で実装する。RAGClient.cs が HTTPブリッジへの呼び出しを担当し、RAGChatWindow.cs がチャットUIを描画する。

C# RAGClient.cs — localhost:8766 呼び出し
public class RAGClient
{
    private const string Endpoint = "http://localhost:8766/query";
    private static readonly HttpClient Http = new();

    public async Task<string> QueryAsync(string query, string namespace_, string apiKey)
    {
        var req = new HttpRequestMessage(HttpMethod.Post, Endpoint);
        req.Headers.Add("X-API-Key", apiKey);
        req.Content = new StringContent(
            JsonConvert.SerializeObject(new { query, namespace_ }),
            Encoding.UTF8, "application/json");

        var res  = await Http.SendAsync(req);
        var json = await res.Content.ReadAsStringAsync();
        return json; // 呼び出し側でチャンク配列をパース
    }
}

Tips: ホットキー登録

SceneView.duringSceneGui コールバックで Shift+Q を検知してウィンドウを開くと、作業中に素早くアクセスできる。

Houdini Python UIパネル（PySide6）

Houdiniの組み込みPythonには requests が入っていない環境が多いため、標準ライブラリの urllib.request のみで実装する。UIブロックを避けるため別スレッドでHTTPリクエストを行い、Qtシグナルで結果をメインスレッドに戻す。

Python panel.py — localhost:8766 呼び出し
import hou
import urllib.request, json, threading
from PySide6 import QtWidgets, QtCore

BRIDGE_ENDPOINT = "http://localhost:8766/query"

class RAGChatPanel(QtWidgets.QDialog):
    answer_received = QtCore.Signal(str)

    def __init__(self):
        super().__init__(hou.qt.mainWindow())
        self.setWindowTitle("RAG Chat — Houdini")
        self.answer_received.connect(self._on_answer)

    def _send(self, query, namespace, api_key):
        threading.Thread(
            target=self._fetch, args=(query, namespace, api_key), daemon=True
        ).start()

    def _fetch(self, query, namespace, api_key):
        try:
            payload = json.dumps({"query": query, "namespace": namespace}).encode()
            req = urllib.request.Request(
                BRIDGE_ENDPOINT, data=payload,
                headers={"Content-Type": "application/json",
                         "X-API-Key": api_key}
            )
            with urllib.request.urlopen(req, timeout=30) as r:
                data = json.loads(r.read().decode())
                self.answer_received.emit(json.dumps(data, ensure_ascii=False))
        except Exception as e:
            self.answer_received.emit(f"エラー: {e}")

    def _on_answer(self, text):
        # チャットビューに描画
        pass

外部ライブラリ不要の理由

urllib.request は標準ライブラリなので追加インストール不要でどのHoudiniでも動く。シェルフツールに import rag_chat; rag_chat.show() を登録するだけで呼び出せる。

タイムアウト設定

埋め込み生成＋ChromaDB検索の処理時間は通常数百ms〜数秒だが、初回起動時はモデルロードが入るため長くなる。各クライアントのHTTPタイムアウトは30秒程度を確保すること。

Troubleshooting

トラブルシューティング

uv sync で sentencepiece のビルドが失敗する

pyproject.toml の requires-python が >=3.13 になっているか確認する。Python 3.13 以降であれば pre-built wheel が提供されているためビルド不要。

'NoneType' object has no attribute 'get_or_create_collection'

vector_database.py の initialize_database() が self.connect() を呼んでいるか確認する（scripts/vector_database.py は変更していなければ問題ない）。

ChromaDB コレクション名エラー（Got: C:\）

SOURCE_DIR 環境変数が正しく設定されていない。

  • SOURCE_DIR が絶対パスで設定されているか確認（デフォルトでは localRAG/ が使われる）
  • 設定後は .processed\ キャッシュを削除して再インデックス
PowerShell
PS> Remove-Item -Recurse -Force localRAG\_rag_dashboard\.processed
PS> uv run python scripts\rag_cli.py index

watchdog が反応しない

PowerShell
# uv run 経由で起動されているか確認
PS> Get-Process python

# 止まっていれば再起動
PS> Start-Process -NoNewWindow -FilePath "uv" -ArgumentList "run python scripts/auto_index.py"

Claude Desktop で MCP が running にならない

  • claude_desktop_config.json のパスが正しいか確認（\\ でエスケープ）
  • PowerShell で uv run --directory "C:\...\DevelopmentRAGEnvironment" python scripts/rag_mcp_server.py が単体で動くか確認
  • Claude Desktop をタスクトレイから完全終了して再起動

インデックス化で _rag_dashboard が処理される

  • _namespace_from_path() が SOURCE_DIR 環境変数を読んで相対パスに変換しているか確認

検索時に KeyError: 'chunk_index' / KeyError: 'document_id' が出る

scripts/vector_database.py の search() / get_adjacent_chunks() / get_document_by_file_path() がこれらのキーをトップレベルで返しているか確認する。

日本語クエリが文字化け / TextEncodeInput エラー（MCP 登録時のみ）

scripts/rag_mcp_server.py の冒頭に UTF-8 パッチ（io.TextIOWrapper）が含まれているか確認する。Windows のデフォルトエンコーディングは CP932 のため、これがないと stdin/stdout で日本語が文字化けする。

エラー例
received query: '繝輔か繝ｼ繝...'   ← 文字化けしている
TextEncodeInput must be Union[TextInputSequence, ...]

インデックス化後も検索結果が増えない（HNSW キャッシュ）

ChromaDB の HNSW インデックスはメモリにロードされる。別プロセスが upsert しても起動中のサーバー側には反映されない。

インデックス化後は必ず MCP サーバー / ブリッジを再起動する。
Claude Desktop: タスクトレイから完全終了 → 再起動
rag_local_bridge.py: プロセスを再起動

全部クリアしてやり直したい（完全リセット）

data\chroma だけでなく file_registry.json（差分インデックスの管理ファイル）も削除しないと、再インデックス時に「差分なし」と判定されファイルが再取り込みされない点に注意。

PowerShell — 完全リセット手順
# 1. ChromaDBデータを削除
PS> Remove-Item -Recurse -Force "data\chroma"

# 2. file_registry.json（差分管理ファイル）を削除
PS> Remove-Item -Force "localRAG\_rag_dashboard\.processed\file_registry.json"

# 3. フルインデックスで再構築
PS> uv run python scripts\rag_cli.py index
