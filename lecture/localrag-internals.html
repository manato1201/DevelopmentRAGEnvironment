section 02

インデックスパイプライン

ファイルをChromaDBに格納するまでの全工程です。RAGService.index_documents()が中心で、DocumentProcessor・EmbeddingGenerator・VectorDatabaseを順番に呼び出します。

1
ディレクトリスキャン document_processor.py

source_dir配下を再帰的にスキャン。最上位フォルダ名がNamespace（ChromaDBコレクション名）になります。_または.で始まるフォルダは除外されます。

対応ファイル形式: .md / .txt / .pptx / .docx / .pdf

markitdown変換

.pptx / .docx / .pdf はmicrosoft/markitdownで先にMarkdown変換してからチャンク分割されます。

2
差分チェック incremental mode

各ファイルに対してSHA-256ハッシュ・mtime・sizeを計算し、file_registry.jsonの記録と比較します。変更がなければスキップ。新規・変更ファイルのみ処理します。

3
チャンク分割 split_into_chunks()

テキストを500文字ずつに分割し、前後100文字のオーバーラップを持たせます。文の途中で切れないよう、\nまたは。を探して文境界で調整します。

4
エンベディング生成 embedding_generator.py

全チャンクテキストに "passage: " プレフィックスを付加し、intfloat/multilingual-e5-largeモデルで1024次元ベクトルに変換。バッチ処理で効率化されています。

5
ChromaDB格納 vector_database.py

batch_insert_documents()でチャンクテキスト・ベクトル・メタデータをまとめてChromaDBに書き込みます。ChromaDBは ./data/chroma/ にファイルとして永続化されます。HNSWインデックスはメモリ上で高速検索できる状態になります。

Python rag_service.py — index_documents()
# チャンク → エンベディング → DB挿入 の流れ
chunks = document_processor.process_directory(source_dir, ...)
texts = [chunk["content"] for chunk in chunks]
embeddings = embedding_generator.generate_embeddings(texts)   # "passage: " prefix

for i, chunk in enumerate(chunks):
    documents.append({
        "document_id": chunk["document_id"],
        "content":     chunk["content"],
        "embedding":   embeddings[i],   # 1024次元ベクトル
        "metadata":    { "file_name": ..., "directory_suffix": ... },
    })

vector_database.batch_insert_documents(documents)
section 03

チャンク分割の詳細

ドキュメントを適切な大きさに切り出すのがRAG精度の根幹です。大きすぎると無関係な情報が混入し、小さすぎると文脈が失われます。

分割アルゴリズム

Python document_processor.py — split_into_chunks()
def split_into_chunks(self, text, chunk_size=500, overlap=100):
    chunks = []
    start = 0
    while start < len(text):
        end = start + chunk_size
        if end < len(text):
            # 文境界を探す（改行 or 句点）
            next_newline = text.find("\n", end)
            next_period  = text.find("。", end)
            # より近い方で切る（見つからなければchunk_sizeで強制分割）
            end = min(next_newline, next_period) + 1
        chunks.append(text[start:end])
        # 次のチャンクはoverlap分だけ戻って開始
        start = end - overlap if end - overlap > start else end

オーバーラップの効果

隣接チャンク間で100文字を共有することで、チャンク境界をまたぐ文脈が失われません。検索ヒットしたチャンクの前後が自然に繋がります。

Chunk 0 (0〜500字)
Chunk 1 (400〜900字)
Chunk 2 (800〜1300字)
Chunk 3 (1200〜…)

薄紫の領域がオーバーラップ部分（前後100字を共有）

トピック単位ファイル分割の重要性

1ファイルに複数トピックを混在させるとスコア差が消える

例: 1ファイルにアイアン・ウェッジ・アパレル情報が混在 → 全チャンクが「FORTEENブランド」として均一にエンコードされ、スコアが81〜83%に団子状態になる。トピックごとにファイルを分割すると、チャンクが専門的な意味空間に集中し、スコア差が拡大して精度が向上します。

section 04

エンベディング生成

テキストを数値ベクトルに変換することで、「意味の近さ」を距離として計算できるようになります。このRAGでは intfloat/multilingual-e5-large を使用します。

📐 モデルスペック
  • 出力次元: 1024次元
  • 日英バイリンガル対応
  • 最大トークン長: 512 tokens
  • HuggingFace Hub から自動ダウンロード
🔑 必須プレフィックス
  • ドキュメント: "passage: " + text
  • クエリ:      "query: " + text
  • 付けないと精度が大幅低下
  • .env の EMBEDDING_PREFIX_* で制御

プレフィックスの使い分け

Python embedding_generator.py
# ドキュメントをインデックス化するとき
def generate_embeddings(self, texts):
    prefixed = ["passage: " + t for t in texts]   # ← passage: プレフィックス
    return model.encode(prefixed, normalize_embeddings=True)

# ユーザーのクエリを検索するとき
def generate_search_embedding(self, query):
    prefixed = "query: " + query                   # ← query: プレフィックス
    return model.encode(prefixed, normalize_embeddings=True)

なぜプレフィックスが必要？

e5シリーズはcontrastive learningで学習されており、「passage:」と「query:」の非対称な関係を学習しています。同じプレフィックスで両方を計算すると類似度が低下します。

類似度スコアの見方

コサイン類似度（0〜1）

ChromaDBはL2距離で検索し、1 - (distance / 2)でコサイン類似度に変換して返します。85%以上が「関連性が高い」の目安。80〜85%はノイズが混入し始めるゾーンです。トピック分割によりスコア差が広がり、精度が向上します。

Python rag_service.py — search()
def search(self, query, limit=5, with_context=False, ...):
    # 1. クエリを "query: " プレフィックス付きでベクトル化
    query_embedding = embedding_generator.generate_search_embedding(query)

    # 2. ChromaDBでHNSW近傍探索
    results = vector_database.search(query_embedding, limit)

    # 3. 前後チャンクを追加取得（with_context=True のとき）
    if with_context:
        adjacent = vector_database.get_adjacent_chunks(file_path, chunk_index, ...)
        results = merge_and_sort(results, adjacent)

    return results  # [{content, similarity, file_path, chunk_index}, ...]

section 07

差分インデックス（増分更新）

毎回全ファイルを処理すると時間がかかります。incremental=Trueを指定すると変更ファイルのみを処理し、大幅に高速化できます。

A
ファイル情報を計算

対象ファイルの SHA-256ハッシュ・mtime（更新日時）・size（バイト数） を計算します。

B
file_registry.json と照合

data/file_registry.jsonに前回インデックス時の情報が記録されています。3値がすべて一致したらスキップ。1つでも変わっていれば再処理します。

C
変更ファイルのみ再インデックス

旧チャンクをChromaDBから削除 → 新チャンクを生成して挿入 → file_registry.jsonを更新します。

.env 差分インデックスの設定
# scripts/index_documents.py 実行時のオプション
PS> python scripts/index_documents.py --incremental

# watchdog (auto_index.py) は常に incremental=True で動作
🕐 フルインデックス

全ファイルを処理。初回セットアップ時や全再構築時に使用。342チャンクで約140秒。

⚡ 差分インデックス

変更ファイルのみ処理。1〜3ファイルなら数秒で完了。watchdogが自動実行。

Claude DesktopとmcpサーバーはModel Context Protocol (MCP) のstdio transport で通信します。プロセス間をstdin/stdoutで繋ぎ、JSON-RPC 2.0形式のメッセージをやり取りします。

Windows CP932問題とパッチ

Windows標準の文字コードはCP932（Shift-JIS）

Windowsのstdoutはデフォルトでcp932エンコードを使います。日本語文字やUnicode文字が含まれるJSONをstdoutに書くと文字化けしてMCPが壊れます。manato1201フォークではmain.pyにUTF-8強制パッチが当たっています。

Python main.py — UTF-8パッチ
import sys, io
# stdoutをUTF-8に強制（Windows CP932問題の回避）
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")
sys.stderr = io.TextIOWrapper(sys.stderr.buffer, encoding="utf-8")
