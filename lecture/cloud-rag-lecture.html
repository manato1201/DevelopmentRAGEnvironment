v1（キーワード検索）との違い

v1はNotionにキーワードクエリを投げて summary を取得していた。v2は全ページ本文をEmbeddingでベクトル化しGoogle Sheetsに保存。クエリ時はコサイン類似度でチャンク単位検索するため、キーワードが含まれなくても意味的に近いドキュメントが取得できる。

チャンク分割（LocalRAGと統一）

LocalRAG（ChromaDB）と同じ設定でチャンク分割することで、両システムの情報量を揃える：

Code.gs — chunkText_
function chunkText_(text, size, overlap) {
  size    = size    || 350;
  overlap = overlap || 70;
  if (!text || text.length <= size) return text ? [text] : [];
  var chunks = [], start = 0;
  while (start < text.length) {
    chunks.push(text.substring(start, start + size));
    if (start + size >= text.length) break;
    start += size - overlap;  // オーバーラップ分だけ戻る
  }
  return chunks;
}

// 呼び出し: LocalRAGに合わせて 500文字 / 100文字オーバーラップ
var chunks = chunkText_(full, 500, 100);

CacheServiceによるインデックスキャッシュ

Sheetsの全行読み込みはリクエストごとに発生すると遅い。GASの CacheService を使い、インデックス全体を6時間キャッシュする。シートの上限100KBを超えるため90KB単位で分割保存する：

Code.gs — CacheService（90KB分割）
var IDX_CACHE_KEY = 'rag_idx_v2';
var CACHE_TTL     = 21600;   // 6時間
var CACHE_CHUNK   = 90000;   // 90KB/チャンク（上限100KB）

function saveIndexCache_(idx) {
  var cache = CacheService.getScriptCache();
  var json  = JSON.stringify(idx);
  var parts = {};
  var n     = Math.ceil(json.length / CACHE_CHUNK);
  for (var i = 0; i < n; i++)
    parts[IDX_CACHE_KEY + '_' + i] = json.substring(i * CACHE_CHUNK, (i+1) * CACHE_CHUNK);
  parts[IDX_CACHE_KEY + '_n'] = String(n);
  cache.putAll(parts, CACHE_TTL);
}

function loadIndex_() {
  var cache = CacheService.getScriptCache();
  var n = parseInt(cache.get(IDX_CACHE_KEY + '_n') || '0', 10);
  if (n > 0) {
    /* キャッシュHIT: チャンクを結合してJSONパース */
    var keys = [];
    for (var i = 0; i < n; i++) keys.push(IDX_CACHE_KEY + '_' + i);
    var vals = cache.getAll(keys);
    var json = '', ok = true;
    for (var i = 0; i < n; i++) {
      if (!vals[IDX_CACHE_KEY + '_' + i]) { ok = false; break; }
      json += vals[IDX_CACHE_KEY + '_' + i];
    }
    if (ok) try { return JSON.parse(json); } catch(e) {}
  }
  return loadIndexFromSheet_();  // キャッシュMISS: シートから読む
}

Architecture

並列同期の実装（fetchAll）

GAS の UrlFetchApp.fetchAll() は複数HTTPリクエストを並列発行できる。同期処理の3フェーズすべてで使用することで、大幅に高速化している。

フェーズ
旧（直列）
新（並列）
DB ページ一覧取得（7DB）
7回×直列
fetchAll 1回
本文取得（更新ページN件）
N回×直列+sleep
fetchAll 1〜2回
Embedding生成（チャンクM件）
1件ずつ+sleep
fetchAll 10件バッチ

20ページ更新・60チャンクの場合: ~22秒 → ~2秒（理論値）

Code.gs — Phase1 並列DB取得
// Phase 1: 全DBのページ一覧を1回の fetchAll で並列取得
var listReqs = [];
Object.keys(DB_KEY_MAP).forEach(function(key) {
  var dbId = props.getProperty(DB_KEY_MAP[key]);
  listReqs.push({
    url:         'https://api.notion.com/v1/databases/' + dbId + '/query',
    method:      'post',
    headers:     nHeaders,  // getNotionHeaders_() を1回だけ呼んでおく
    contentType: 'application/json',
    payload:     JSON.stringify({ page_size: 100 }),
    muteHttpExceptions: true,
  });
});
var listResps = UrlFetchApp.fetchAll(listReqs);  // ← ここが並列

// Phase 3: Embedding を 10件バッチで並列取得
var BATCH_SIZE = 10;  // Gemini レートリミット対策
for (var b = 0; b < allChunks.length; b += BATCH_SIZE) {
  var batch     = allChunks.slice(b, b + BATCH_SIZE);
  var embedReqs = batch.map(function(c) { return { url: embedUrl, ... }; });
  UrlFetchApp.fetchAll(embedReqs).forEach(function(res, j) {
    var emb = JSON.parse(res.getContentText()).embedding.values;
    newRows.push([c.page_id + '::' + c.k, c.db, c.title, c.text, c.last_edited, JSON.stringify(emb)]);
  });
  if (b + BATCH_SIZE < allChunks.length) Utilities.sleep(200);
}

クエリ時は並列化不要

検索時（searchByEmbedding_）はインデックスをメモリ展開してコサイン類似度を一括計算するため、DBごとに分ける処理がない。並列化の恩恵はSync処理のみ。

Architecture

マルチターン会話（ragQuery）

Gemini API の contents 配列を使い、過去の会話履歴を含めてマルチターン対話を実現する。knowledge baseの注入は最初のuser/modelペアとして挿入する。

Code.gs — ragQuery（マルチターン）
function ragQuery(query, dbKey, history) {
  history = history || [];

  // ① ベクトル検索で上位5チャンク取得
  var results = searchByEmbedding_(query, dbKey, 5);
  var context = results.length === 0
    ? '（関連ドキュメントが見つかりませんでした）'
    : results.map(function(r, i) {
        return '### [' + (i+1) + '] ' + r.title + '\n' + r.text;
      }).join('\n\n');

  // ② contents 配列を構築
  //   [知識ベース注入(user/model)] + [会話履歴] + [今回の質問]
  var contents = [
    { role: 'user',  parts: [{ text: '以下の参考ドキュメントを確認しました。\n\n' + context }] },
    { role: 'model', parts: [{ text: '参考ドキュメントを確認しました。ご質問にお答えします。' }] },
  ];

  // ③ 直近3往復（6メッセージ）を追加
  history.slice(-6).forEach(function(h) {
    contents.push({ role: h.role === 'bot' ? 'model' : 'user', parts: [{ text: h.text }] });
  });
  contents.push({ role: 'user', parts: [{ text: query }] });

  // ④ Gemini 呼び出し（system_instruction で回答スタイルを制御）
  var answer = callGemini_(contents);
  return { answer: answer, sources: deduplicateSources_(results) };
}

function callGemini_(contents) {
  var payload = JSON.stringify({
    system_instruction: { parts: [{ text:
      'あなたはゲーム開発チームの知識ベースを持つAIアシスタントです。\n' +
      '日本語で**簡潔に**回答してください（目安: 400文字以内）。\n' +
      '重要な点のみ箇条書き（-）または短い見出し（##）でまとめてください。'
    }]},
    contents:          contents,
    generationConfig:  { temperature: 0.4, maxOutputTokens: 1024 },
  });
  // ... Gemini API 呼び出し
}

トークン節約のポイント

会話履歴は直近3往復（6メッセージ）のみ保持。本文テキストはチャンクを600字に絞る。maxOutputTokens: 1024で出力を制限。これにより入力4K・出力7Kまで増加していたトークンを大幅削減。

Step 05

チャットUIの実装

doGet() でHTMLを返す。google.script.run でサーバーサイドの ragQuery を呼び出す構成。非同期であるためAPIキーがブラウザに露出しない。

重要: window.history との変数名衝突

⚠️ window.history バグ

ブラウザのJSでは var history = [] と書くと、ブラウザのNavigation History API（window.history）を上書きしてしまう。history.slice() が TypeError を投げ、isSending=true のままUIがロックされる。変数名は必ず chatHistory にすること。

HTML — 会話履歴変数（正しい命名）
// ❌ NG: window.history と競合する
var history = [];

// ✅ OK: chatHistory に命名
var chatHistory = [];

function clearChat() {
  chatHistory = [];  // ← chatHistory を使う
  /* ... */
}

function send() {
  var historySnapshot = chatHistory.slice();  // ← コピーして渡す
  google.script.run
    .withSuccessHandler(function(result) {
      chatHistory.push({ role: 'user', text: q });
      chatHistory.push({ role: 'bot',  text: result.answer });
    })
    .ragQuery(q, dbKey, historySnapshot);
}

Troubleshooting

トラブルシューティング

Notion APIが403を返す

インテグレーションが各DBの「接続先」に追加されているか確認する。Notion DB → 右上「…」→「接続先」→ インテグレーションを追加。

syncNotionToSheets がエラー

SHEETS_ID がスクリプトプロパティに設定されているか確認。スプレッドシートのURLからIDを正しくコピーしているか確認。

GAS — Embedding疎通確認
// testEmbedding() を実行して確認
✅ embedding-001 OK  次元数: 768  先頭3値: 0.012,0.034,-0.011

// ❌ の場合: GEMINI_API_KEY を確認する

UIが「応答中」のまま固まる

var history = [] が window.history（ブラウザの Navigation API）と競合している。history.slice() が TypeError を投げ、isSending=true のままUIがロックされる。

解決: getChatHtml() 内の変数名をすべて chatHistory に変更して再デプロイ（現行コードでは修正済み）。

検索精度が低い / 関連ドキュメントが見つからない

  • Notionページの summary プロパティに内容を追記する（空だと検索精度が落ちる）
  • 固有名詞（BTQ-116、TC-8 等）をページタイトルや summary に明示する
  • syncNotionToSheets を再実行してインデックスを最新化する
  • testSearch('質問文', 'braintq', 3) で検索結果をログ確認する

同期が途中で止まる（タイムアウト）

GASの実行時間制限（6分）を超えた場合。並列化で大幅改善済みだが、超過する場合は DB_KEY_MAP を一時的に絞って分割実行する。

Notionコードブロックが 400エラー

localrag_to_notion.py でコード言語の指定が Notion allowlist に含まれない場合（例: csharp は無効、c# が正しい）。_NOTION_LANG マッピングで自動変換されるが、マッピング外の言語は plain text にフォールバックする。
