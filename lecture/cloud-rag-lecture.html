Chapter 01

RAG とは何か — AI が「本を見ながら答える」仕組み

RAG（Retrieval-Augmented Generation、検索拡張生成）は、AI が質問に答えるときに あらかじめ用意した知識データベースを検索してから回答を生成する手法です。 「参考書を見ながら試験に答えていい」方式と考えてください。

ChatGPT との違い

通常の ChatGPT
  • 学習済みの知識のみで回答
  • 知識のカットオフ日以降は知らない
  • 自社・個人の非公開情報は一切知らない
  • 「幻覚（ハルシネーション）」で嘘をつく
RAG システム
  • 独自ドキュメントから検索して回答
  • データを追加すればリアルタイムに更新可能
  • Notion に書いた社内仕様書も参照できる
  • 「根拠」となるドキュメントを提示できる

RAG の 4 ステップ

1
ドキュメントをベクトル化して保存 インデックス構築

Notion のページや Markdown ファイルを AI に渡すと、各文章が 「数値の配列（ベクトル）」 に変換されてデータベースに保存されます。これを インデックス構築 と呼びます。

2
質問もベクトル化 クエリ

「Unity でカメラを揺らすには？」という質問も同じ方法でベクトルに変換します。

3
最も「似ている」ドキュメントを取得 Retrieval（検索）

質問ベクトルとドキュメントベクトルを比較し、コサイン類似度（後述）で上位 5 件を取得します。

4
LLM に「文脈 + 質問」を渡して回答生成 Generation（生成）

取得したドキュメントを「参考情報」として Gemini などの LLM に渡し、根拠のある回答を生成させます。

ベクトル（数値の配列）とは？

「意味」を数値で表す考え方です。例えば multilingual-e5-large モデルは 文章を 1024 個の数値（1024 次元ベクトル）に変換します。 意味が似ている文章は、ベクトル空間で近い位置に配置されます。 


  「Unity でシェーダーを書く方法」 → [0.021, -0.043, 0.187, ..., 0.032]  (1024次元)
  「HLSL でグラフィクスプログラミング」 → [0.019, -0.040, 0.183, ..., 0.031]  ← 近い！
  「ラーメンのレシピ」              → [0.891,  0.231, -0.045, ..., 0.442]  ← 遠い

コサイン類似度 — 「角度」で似ているかを測る

2 つのベクトルの方向が揃っているほど（角度が小さいほど）「意味が似ている」と判断します。 コサイン類似度は -1.0〜+1.0 の値で、1.0 に近いほど意味が一致します。

↑ ベクトルAとBの角度が小さい（コサイン類似度が高い）ほど「意味が似ている」

コサイン類似度意味このシステムでの扱い
0.90〜1.00ほぼ同じ意味トップヒット
0.75〜0.90かなり関連あり上位5件に含まれる
0.70〜0.75関連ありグラフビューの「エッジ」として表示
0.70未満あまり関連なし検索結果に含めない

数学的な定義

コサイン類似度 = cos(θ) = (A · B) / (|A| × |B|)。ドット積を各ベクトルの長さで割ったもの。ベクトルの長さ（大きさ）ではなく「向き」だけを比較するため、文章の長さが異なっても公平に比較できます。

埋め込み（Embedding）とは

テキストを 高次元の数値ベクトル に変換する処理です。 このプロジェクトでは gemini-embedding-001 を使い、入力文章を 768 次元のベクトルに変換します。 768 個の数値が文章の「意味の座標」として機能します。

GAS — Gemini Embedding API 呼び出し
// gemini-embedding-001 に文章を渡して 768次元ベクトルを取得
function getEmbedding_(text) {
  var url = 'https://generativelanguage.googleapis.com/v1beta/models/'
          + 'gemini-embedding-001:embedContent?key=' + GEMINI_KEY;
  var res = UrlFetchApp.fetch(url, {
    method: 'post',
    contentType: 'application/json',
    payload: JSON.stringify({
      model: 'models/gemini-embedding-001',
      content: { parts: [{ text: text }] }
    })
  });
  return JSON.parse(res.getContentText()).embedding.values;
  // → [0.012, -0.034, 0.187, ..., 0.041]  768個の数値配列
}

チャンク分割 — なぜ文章を細かく切るのか

LLM の入力には トークン制限があります。また、長い文章をそのままベクトル化すると 「意味が薄まり」検索精度が下がります。そのため文章を適切なサイズに分割（チャンク）して インデックスに登録します。

このシステムのチャンク設定

チャンクサイズ: 500 文字（約 3〜4 センテンス）
オーバーラップ: 100 文字（前のチャンクの末尾 100 文字を次のチャンク冒頭にも含める）
→ 文章が途中で切れても前後の文脈が繋がる。Cloud RAG と Local RAG で同一設定を使用。 


  元文章: 「AAABBBCCCDDDEEEFFFGGGHHHIIIJJJKKKLLLMMMNNNOOOppp...（800文字）」

  チャンク1: AAABBBCCCDDDEEEFFFGGG...（500文字）
  チャンク2:       DDDEEEFFF...（オーバーラップ100文字から開始・次の500文字）
  チャンク3:             GGG...

  → 境界部分も検索でヒットするように重複させる

HyDE 検索強化 — 仮説ドキュメントで語彙ミスマッチを解消

通常の RAG では「ユーザーの質問」をそのまま埋め込んで検索します。 しかし質問の語彙（「Unity でカメラを揺らすには？」）とドキュメントの語彙（「CinemachineImpulseSource の実装例」）が かけ離れていると、コサイン類似度が低くなり正しいドキュメントがヒットしません。 HyDE（Hypothetical Document Embedding） はこの語彙ミスマッチを解決する手法です。

1
ユーザーの質問を受け取る

例：「Unity でカメラを揺らすには？」

2
ドメイン特化プロンプトを選択 hydePromptFor_(dbKey)

hydePromptFor_(dbKey) が DB キーに応じた専門分野プロンプトを返します。例えば tool_docs なら「あなたは Unity/Houdini のベテランエンジニアです。以下の質問に対して実際のコードやAPIを含む3〜5文の技術的な回答を書いてください。」という指示になります。

3
仮説的回答ドキュメントを生成 Gemini で生成

Gemini が「CinemachineImpulseSource を GameObject にアタッチし、GenerateImpulse() を呼び出します…」のような 3〜5 文の仮説的回答を生成します。これはユーザーへ返す回答ではなく、あくまで検索精度向上のための内部的な「架空の正解文書」です。

4
仮説ドキュメントを埋め込み → 重み付き平均 query×0.4 + hypothesis×0.6

質問と仮説ドキュメントをそれぞれ 768 次元ベクトルに変換し、加重平均で最終クエリベクトルを計算します。仮説ドキュメントのほうが実際のドキュメント語彙に近いため、重みを 60% にします。

5
閾値フィルタ → ページ重複排除 Retrieval

平均ベクトルでインデックスを検索し、DB 個別閾値 0.58 / 全 DB 検索時 0.62 でフィルタリング。同一ページから複数チャンクがヒットした場合は最高スコアのチャンクのみを残します（ページ重複排除）。

HyDE の利点と注意点

利点：ユーザーが専門用語を知らなくても、AI が架空の正解文書を「翻訳者」として機能させることで検索精度が向上します。

Chapter 04

Local RAG の仕組み — ChromaDB + Claude Haiku

Local RAG はインターネットに接続せず、自分のマシンだけで動く RAG システムです。 個人情報やプライベートなメモを安全に検索できます。

mcp-rag-server とは

mcp-rag-server は MCP（Model Context Protocol） サーバーです。 JSON-RPC over stdio という通信方式で Claude Desktop と接続し、 「ドキュメントを検索する」ツールを Claude に提供します。

MCP とは？

Anthropic が策定したオープンプロトコルで、LLM（Claude など）に外部ツールを接続する標準的な方法です。「Claude にデータベース検索能力を追加する」ための仕組みと考えてください。stdio（標準入出力）経由で JSON を送受信します。

ChromaDB — ローカルのベクトルデータベース

ChromaDB は Python で書かれたオープンソースのベクトルデータベースです。 インストールするだけでローカルでベクトル検索が使えます。 ファイルとして保存されるため、Docker 不要・サーバー不要で動作します。

ChromaDB の特徴
  • pip install chromadb だけでOK
  • ファイルシステムに永続化
  • Python API から直接操作
  • BM25 + ベクトル RRF ハイブリッド検索
  • メタデータフィルタリング対応
multilingual-e5-large
  • Microsoft 製の多言語埋め込みモデル
  • 日本語に強い（Multilingual）
  • 1024 次元ベクトルを生成
  • ローカル実行可能（API 不要）
  • SentenceTransformers 経由で使用

rag_local_bridge.py — FastAPI なしの HTTP サーバー

Unity や Houdini から Local RAG に接続するための HTTP ブリッジサーバーです。 FastAPI などのフレームワークを使わず、Python 標準ライブラリの http.server だけで実装されています。

なぜ標準ライブラリだけで書いたのか？

追加 pip install が不要になるため、チームメンバーの環境セットアップが簡単になります。FastAPI は高機能ですが、依存関係の衝突リスクがあります。RAG ブリッジに必要な 3 エンドポイントは標準ライブラリで十分実装できます。

# /query エンドポイントの処理（簡略版）
def handle_query(body):
    query_text = body['query']

    # 1. mcp-rag-server に検索を依頼（JSON-RPC over stdio）
    results = mcp_search(query_text, n_results=5)

    # 2. 取得したドキュメントを文脈として組み立て
    context = '\n\n'.join([r['text'] for r in results])

    # 3. Claude Haiku で回答生成
    answer = anthropic_client.messages.create(
        model='claude-haiku-4-5',
        messages=[{
            'role': 'user',
            'content': f'参考情報:\n{context}\n\n質問: {query_text}'
        }]
    ).content[0].text

    return { 'answer': answer, 'sources': results }
Chapter 05

グラフビュー — ドキュメント間の「隠れた関係」を可視化

ドキュメントが増えると「どの文書が関連しているか」がわかりにくくなります。 グラフビューは コサイン類似度が 0.70 以上のドキュメント同士をエッジ（線）で結ぶ ことで、知識の繋がりをネットワーク図として可視化します。

なぜグラフが必要か

テキスト検索の限界

「BrainTQ の AI 設計」を検索しても、「強化学習の実装」という別ページとの関連が見えない。

グラフビューの価値

コサイン類似度 0.70 以上のページが自動的にエッジで結ばれるため「意味的に近い文書群」が一目でわかる。

コサイン類似度 0.70 のエッジ定義


  全ドキュメントペアのコサイン類似度を計算:

  doc_A × doc_B → similarity = 0.83  ✅ エッジを張る
  doc_A × doc_C → similarity = 0.71  ✅ エッジを張る
  doc_A × doc_D → similarity = 0.65  ❌ エッジを張らない（閾値 0.70 未満）

  閾値を下げすぎると → ほぼ全ノードが繋がり「地図」にならない
  閾値を上げすぎると → 孤立ノードが増え「孤島」だらけになる
  0.70 は実験的に最も「意味のある関係」が見える値として採用

Spring レイアウト（力学モデル配置）

ノードの配置は バネ-電荷モデル（Force-Directed Layout） を使います。 繋がっているノード同士はバネで引き合い、繋がっていないノードは電荷で反発し合うことで、 自然に「クラスター（塊）」が形成されます。

物理シミュレーションによる自動配置

人間が手動でノードを配置する必要がありません。「意味的に似たドキュメント群」が自動的に近くに集まります。これが Spring Layout（力学モデル）の特徴です。Networkx の spring_layout() が同じアルゴリズムを使用しています。

rag_graph_export.py — コサイン類似度行列の計算
import numpy as np

def build_graph(embeddings, titles, threshold=0.70):
    # numpy で一括計算（ループなし・高速）
    norms = np.linalg.norm(embeddings, axis=1, keepdims=True)
    normalized = embeddings / (norms + 1e-10)

    # 全ペアのコサイン類似度行列（N×N）を一気に計算
    sim_matrix = normalized @ normalized.T  # ドット積 = コサイン類似度

    edges = []
    n = len(titles)
    for i in range(n):
        for j in range(i + 1, n):
            if sim_matrix[i, j] >= threshold:
                edges.append({
                    'source': i, 'target': j,
                    'similarity': float(sim_matrix[i, j])
                })
    return edges

QThread でバックグラウンド通信

Python の requests.post() は同期処理のため、メインスレッドで呼ぶと Houdini 全体が固まります。QThread にオフロードすることで AI が回答を生成している間も Houdini を操作できます。

Python — QThread でバックグラウンド HTTP 通信
from PySide6.QtCore import QThread, Signal

class QueryWorker(QThread):
    finished = Signal(dict)   # 完了時にメインスレッドへ通知
    error    = Signal(str)

    def __init__(self, url, payload):
        super().__init__()
        self.url, self.payload = url, payload

    def run(self):
        # ← このメソッドはバックグラウンドスレッドで実行される
        try:
            res = requests.post(self.url, json=self.payload, timeout=60)
            self.finished.emit(res.json())
        except Exception as e:
            self.error.emit(str(e))

# メインスレッドで Worker を起動
self._worker = QueryWorker(url, payload)
self._worker.finished.connect(self.on_query_done)  # シグナル接続
self._worker.start()  # バックグラウンドで HTTP 通信を開始

import json, pathlib

SETTINGS_PATH = pathlib.Path.home() / '.houdini_rag_settings.json'

def load_settings():
    if SETTINGS_PATH.exists():
        return json.loads(SETTINGS_PATH.read_text())
    return { 'bridge_url': 'http://localhost:8765', 'gas_url': '' }

def save_settings(cfg):
    SETTINGS_PATH.write_text(json.dumps(cfg, ensure_ascii=False, indent=2))

セキュリティ設計 — API キーを絶対に漏らさないために

API キー（Notion・Gemini・Anthropic）は一度 GitHub に Push してしまうと ボットに即座に取得され、不正利用されるリスクがあります。 このシステムでは複数の層でキーを保護しています。

なぜコードに直接書いてはいけないか

API キーをソースコードに書くと → git commit → GitHub Push の瞬間に世界中に公開されます。 GitHub は定期的にキーのパターンをスキャンし、Gemini や Anthropic に自動通報する仕組みもあります。 ただし、それより早くボットがスキャンしてキーを悪用するケースが多いです。

レイヤ別のキー保管場所

API キー保管場所理由
NOTION_API_KEY
GEMINI_API_KEY		 GAS スクリプトプロパティ GAS のサーバーサイドに閉じており、ブラウザや Git には一切出ない
GAS WebApp URL Unity EditorPrefs
Houdini JSON（~/.houdini_rag_settings.json）		 Gitリポジトリに含まれないユーザーローカル領域に保存

環境変数での管理方法

PowerShell — 環境変数の設定（セッション永続）

    "User"
  )

# Python コードで取得
import os
api_key = os.environ["ANTHROPIC_API_KEY"]  # コードに直接書かない

.env ファイルを使わない理由

❌ .env ファイルの問題
  • .gitignore への追記を忘れると即 Push される
  • チームメンバーへの配布が困難
  • 複数プロジェクトで同じキーを使いにくい
  • CI/CD 環境では使えない
✅ OS 環境変数のメリット
  • Git リポジトリに絶対含まれない
  • OS レベルで暗号化されて保存
  • 全プロジェクトで共有できる
  • CI/CD（GitHub Actions の Secrets）と同じ考え方

GAS URL の Git 除外

GAS の WebApp URL はそれ自体がエンドポイントのため、 公開すると誰でも GAS 関数を呼び出せてしまいます（Notion への読み取り権限を悪用される恐れ）。 EditorPrefs は Unity のレジストリストレージに保存され、Assets/ フォルダ外のため Git の管理対象に含まれません。

GAS のアクセス制御

GAS WebApp を「アクセスできるユーザー: 自分のみ」に設定すると、 Google アカウントでの認証が必要になるため、URL が漏れても他者は使えません。 チームで共有する場合は「ドメイン内のユーザー」に限定するか、 URL をランダムなトークンで保護する方法が推奨されます。

リアルタイム同期（Webhook）

現在は手動で syncNotionToSheets() を実行して同期します。Notion の Webhook 機能を使えば、ページを書いた瞬間に自動でインデックスが更新される「リアルタイム RAG」を実現できます。GAS のタイムトリガー（1時間ごと）を設定することで半自動化も可能です。

Cloud RAG 側は現在コサイン類似度のみです。Local RAG（ChromaDB）が採用している BM25 + ベクトル RRF ハイブリッド検索を Cloud RAG にも適用することで、キーワード検索と意味検索を組み合わせた高精度検索が実現できます。特に固有名詞（型番・関数名）の検索精度が向上します。

RAG-Fusion（複数クエリ展開）

1 つの質問から複数の「言い換えクエリ」を自動生成し、各クエリで検索して結果をマージする手法です。「Unity でシェーダーが動かない」という質問から「HLSL コンパイルエラー」「Material PropertyBlock 設定」など複数の角度で検索することで、検索の抜け漏れを減らせます。

Streaming 回答生成

現在は回答が全文生成されてから表示されます。Gemini / Claude の Streaming API を使えば、回答を文字単位でリアルタイムに表示できます（ChatGPT と同じ体験）。Unity IMGUI は Repaint() を繰り返すことで部分的に実装可能です。

Obsidian ← → Notion 自動同期

現在は localrag_to_notion.py で手動移行します。Obsidian の Markdown ファイル変更を監視し、差分を自動で Notion に Push するデーモンを作れば、Obsidian でメモを書くだけで Cloud RAG に自動登録されるフローが実現します。

マルチモーダル検索（画像・動画）

Gemini はテキストと画像を同時に処理できます。Unity のスクリーンショットや Houdini のレンダリング結果を質問に添付し「このシェーダーの問題を教えて」と聞けるマルチモーダル RAG への拡張が技術的に可能です。
