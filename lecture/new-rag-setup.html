section 01

全体フロー（5ステップ）

新しい情報領域をRAGに追加するときは、必ずこの順番で進めます。インデックスより先にドキュメント設計を固めることが精度向上の鍵です。

ステップ作業場所目安時間
1 Namespace設計localRAG/配下にフォルダを作成エクスプローラー / PS2分
2 ドキュメント設計トピック分割の粒度を決める設計作業（頭の中）5〜10分
3 MD作成Frontmatter付きMarkdownを作成Obsidian / VSCode10〜20分
4 インデックス実行index コマンドを叩くPowerShell1〜5分
5 動作確認Claude Desktopで検索テストClaude Desktop2〜3分

以降は自動化できます

auto_index.py（watchdog）を起動しておくと、Markdownを保存するたびに差分インデックスが自動実行されます。追加・編集のたびに手動でインデックスコマンドを叩く必要はありません。

section 02

Namespace設計 — フォルダ構造

ChromaDBのコレクション（検索対象グループ）は localRAG/ 直下のフォルダ名が自動的にNamespaceになります。検索範囲を絞るときの単位でもあります。

フォルダを作成する

localRAG/
├── your_new_namespace/ ← ここに新しいフォルダを作る
│ ├── topic_a.md
│ ├── topic_b.md
│ └── topic_c.md
├── personal_notes/ ← 既存（ゴルフ・個人メモ）
├── game_dev/ ← 既存（BrainTQ等）
├── _rag_dashboard/ ← _ プレフィックス = 除外
└── _templates/ ← _ プレフィックス = 除外
PowerShell フォルダ作成
# 例: Unity関連ドキュメント用のNamespaceを作る
PS> New-Item -ItemType Directory -Path "localRAG\unity_docs"

# Obsidianで作る場合は左パネルの「+フォルダ」ボタンでOK

Namespace命名ルール

ルールOK例NG例
英数字・アンダースコアのみunity_docsUnity Docs（スペース不可）
_ または . で始めないgame_dev_game_dev（除外される）
用途が伝わる名前にするhoudini_vexdocs1（何か不明）
localRAG直下の1階層目のみ有効localRAG/unity_docs/localRAG/docs/unity/（unity/はNamespaceにならない）
section 03

Markdownファイル作成ルール

RAG用ドキュメントは通常のMarkdownですが、YAMLフロントマターを付けることでメタデータとして管理できます。

テンプレート（コピーして使う）

Markdown localRAG/your_namespace/topic_name.md
---
title: ここにドキュメントのタイトル
tags:
  - タグ1
  - タグ2
  - namespace名
rag_indexed: true
created: 2026-06-18
updated: 2026-06-18
---

# メインタイトル（frontmatterのtitleと揃える）

ここに本文を書く。

## セクション1

内容...

## セクション2

内容...

フロントマター各フィールドの説明

フィールド必須説明
title必須ドキュメントのタイトル。検索結果のメタデータに含まれる
tags推奨検索キーワードに関連する語句を列挙。Namespace名も入れると良い
rag_indexed必須true に設定。false のままだとドラフト扱いになる
created任意作成日（YYYY-MM-DD）
updated任意更新日（YYYY-MM-DD）
source_url任意参照元URL（Webから収集した情報の場合）

rag_indexed: true の確認を忘れずに

新規作成時にテンプレートを使わないと rag_indexed: false のままになることがあります。インデックス化しても内容が正式登録されていない「ドラフト状態」になります。作成後は必ず確認してください。

一括でフラグを確認・修正する（PowerShell）

PowerShell false のファイルを洗い出す
PS> Get-ChildItem -Path "localRAG\your_namespace" -Filter "*.md" -Recurse |
    Select-String -Pattern "rag_indexed: false" | Select-Object -ExpandProperty Path
PowerShell 対象フォルダ内を一括 true に更新
PS> Get-ChildItem -Path "localRAG\your_namespace" -Filter "*.md" -Recurse | ForEach-Object {
    $content = [System.IO.File]::ReadAllText($_.FullName, [System.Text.Encoding]::UTF8)
    $updated = $content -replace "rag_indexed: false", "rag_indexed: true"
    [System.IO.File]::WriteAllText($_.FullName, $updated, [System.Text.Encoding]::UTF8)
    Write-Host "Updated:" $_.Name
}
section 04

ドキュメント設計 — 検索精度を上げるコツ

RAGの検索精度はドキュメントの書き方・分割の粒度に強く依存します。ここが最も重要です。

原則: 1ファイル = 1トピック

✓ GOOD — トピック分割

  • unity_shader_basics.md
  • unity_shader_urp.md
  • unity_shader_custom_pass.md
  • unity_particle_system.md
  • unity_animation_rigging.md

✗ BAD — 1ファイルに全部

  • unity_all.md（シェーダー・パーティクル・アニメーション・UIすべて混在）
  • → 全チャンクが「Unityドキュメント」として均一にエンコードされる
  • → 検索スコアが80〜83%に団子状態になり、無関係な内容も同スコアでヒット

理想的なファイルサイズ

文字数（本文）チャンク数の目安評価
〜800文字1〜2チャンク小さすぎ — 文脈が薄く精度が不安定になりやすい
800〜3,000文字2〜8チャンク理想 — 1トピックを充分に説明できる量
3,000〜6,000文字8〜14チャンクやや大きい — サブトピックで分割検討
6,000文字〜14チャンク〜分割推奨 — 必ずトピック分割する

内容の書き方

📝 検索されやすい書き方
  • 質問形式の言葉を本文に含める
    「〜とは」「〜の方法」「〜の違い」など
  • 固有名詞・型名・関数名をそのまま書く
    「OnTriggerEnter」「VFXGraph」など
  • 日英どちらで聞かれても答えられる内容にする
    e5-largeは日英バイリンガル対応
  • 箇条書きより文章を混ぜる
    表や箇条書きだけだと文脈が薄くなる
🚫 精度を下げる書き方
  • 省略形・略語のみで書く
    「TB-3」のみ → 「TB-3 FORGED アイアン」と書く
  • 表・コードのみで本文がない
    意味のある文章が1チャンクに1段落は欲しい
  • 異なるトピックを「その他」でまとめる
    「その他機能まとめ.md」は避ける
  • 同じ情報を複数ファイルに重複して書く
    重複チャンクがスコアを汚染する

実例: BrainTQ分割の考え方

GameType enumをファイル分割の軸にした

1ファイル（braintq_minigame.md）にMemory/Calculation/Spatial/Languageなど8カテゴリのゲームが混在していた。GameType enumのカテゴリを分割軸にして10ファイルに分割した結果、チャンク数 223 → 278 に増加し、カテゴリをまたぐ誤ヒットが減少した。
→ 既存のコード・仕様の分類体系があれば、それを分割軸にすると良い

section 05

インデックス実行

ドキュメントが揃ったらインデックスコマンドを実行します。検索エンジンはこのリポジトリに統合済みのため、外部リポジトリへの移動は不要です。

1
リポジトリのルートで実行する（移動不要）
PowerShell
PS> cd C:\Users\YOUR_USERNAME\Desktop\GameDevelopment\DevelopmentRAGEnvironment
2
フルインデックス実行（初回）

新しいNamespaceを初めてインデックス化するときはフルインデックスを推奨します。

PowerShell フルインデックス
PS> uv run python scripts\rag_cli.py index

# 出力例
インデックス化しています... (source: localRAG/)
342 → 365 ドキュメント、23.4秒
2'
差分インデックス実行（2回目以降）

変更・追加分だけを処理するので高速です。既存のNamespaceに数ファイル追加した場合はこちら。

PowerShell 差分インデックス（--incremental）
PS> uv run python scripts\rag_cli.py index --incremental

# 変更なしのファイルはスキップされる
3 files changed, 18 new chunks added (3.2秒)
3
チャンク数を確認

実行後にチャンク総数が増えていれば正常にインデックス化されています。

PowerShell 現在のチャンク数を確認
PS> uv run python scripts\rag_cli.py count

インデックス内のドキュメント数: 365

特定Namespaceのみ再インデックスしたいとき

PowerShell 対象Namespaceをクリアして再インデックス
# 1. まず全インデックスをクリア（慎重に！）
PS> uv run python scripts\rag_cli.py clear

# 2. フルインデックスで全Namespaceを再構築
PS> uv run python scripts\rag_cli.py index

clearコマンドは全データを削除します

clearを実行するとChromaDB内の全コレクションが削除されます。特定Namespaceだけを消す機能は現在なし。clearした後は必ずフルインデックスを実行してください。

section 06

動作確認 — 検索テスト

インデックス完了後はClaude Desktopで実際に検索して期待した結果が返ることを確認します。

Claude Desktopでの確認方法

A
シンプルなキーワード検索

まず登録したドキュメントのキーワードをそのまま聞いてみます。

Claude Desktopでの質問例
local-rag-serverで「[登録したキーワード]」について検索してください。
チャンクのスコアも教えてください。
B
スコアで精度を判定する
スコアレンジ判定対応
88%〜良好そのまま使える
83〜88%普通許容範囲。精度をさらに上げたければトピック分割を細かく
78〜83%要注意関係ない文書がヒットしている可能性。ファイル粒度を見直す
〜78%NGドキュメント内容か分割粒度に問題あり。設計から見直す
C
スコア差で絞り込み精度を確認

上位5件のスコア差が3%以上あれば絞り込みが効いています。差が1%未満の場合はトピックが混在しているサイン。

例: 良い状態

1位: 91.2%（fourteen_irons.md）、2位: 85.4%（fourteen_brand.md）、3位: 79.1%（fourteen_apparel.md）
→ スコア差が大きく、目的のドキュメントが明確に1位

例: 悪い状態（分割が必要）

1位〜5位: 81〜83%（全て same_large_file.md の異なるチャンク）
→ 1ファイルに複数トピックが混在している。分割が必要

Namespace指定で絞り込み検索

Claude Desktopでの質問例
local-rag-serverで namespace="your_namespace" を指定して
「[キーワード]」を検索してください。
section 07

自動化 — auto_index.py（watchdog）

一度セットアップしたらwatchdogを起動しておくことで、ファイルを保存するたびに差分インデックスが自動実行されます。scripts/auto_index.py はこのリポジトリ内にあるため、コピーは不要です。

1
バックグラウンドで起動
PowerShell
PS> Start-Process -NoNewWindow -FilePath "uv" -ArgumentList "run python scripts/auto_index.py"

# または新しいPowerShellウィンドウで前面起動して確認しながら使う
PS> uv run python scripts\auto_index.py
3
Obsidian で保存 → 自動インデックス

Obsidianでファイルを保存（Ctrl+S）すると watchdog が検知し、変更ファイルのみ差分インデックスが走ります。インデックス化の遅延は数秒以内です。

auto_index.py — バックグラウンド出力
Watching: C:/Users/.../localRAG/
File modified: unity_docs/unity_shader_basics.md
Incremental indexing...
  3 new chunks added (1.8秒)
Watching...
section 08

トラブル対応

!
検索でヒットしない

以下をチェックしてください。

  • rag_indexed: true になっているか
  • インデックスコマンド実行後にチャンク数が増えているか
  • フォルダ名が _ や . で始まっていないか
  • Claude Desktopを再起動（MCPサーバーの再接続）
!
[Errno 22] Invalid argument エラーが出る

_rag_dashboard/.processed/ 配下のパスが長すぎてWindowsの最大パス長を超えています。インデックス本体は成功しているので実害はありません。気になる場合は .processed/ フォルダを定期的に掃除してください。

PowerShell .processed を掃除
PS> Remove-Item -Recurse -Force "localRAG\_rag_dashboard\.processed\*"
!
スコアが全体的に低い（〜78%）

原因は大抵どちらかです。

  • クエリとドキュメントの言語が合っていない（e5-largeは日英どちらでも対応できるが、混在するとスコアが下がることがある）
  • ドキュメントの内容が抽象的すぎる。固有名詞や具体的な説明を増やす
  • 対象の情報がそもそもインデックスされていない。チャンク数を確認する
!
全部クリアしてやり直したい
PowerShell 完全リセット手順
# 1. ChromaDBデータを削除
PS> Remove-Item -Recurse -Force "data\chroma"

# 2. file_registry.json（差分管理ファイル）を削除
PS> Remove-Item -Force "localRAG\_rag_dashboard\.processed\file_registry.json"

# 3. フルインデックスで再構築
PS> uv run python scripts\rag_cli.py index

新規RAG追加チェックリスト

  • localRAG/配下に新しいNamespaceフォルダを作成した
  • フォルダ名が英数字・アンダースコアのみで、先頭が _ / . でない
  • トピック単位でファイルを分割した（1ファイル = 1トピック）
  • 各ファイルに YAMLフロントマターを付けた
  • rag_indexed: true を全ファイルに設定した
  • インデックスコマンドを実行してチャンク数の増加を確認した
  • Claude Desktopで検索テストしてスコア83%以上を確認した
  • auto_index.py を起動して以降の自動化を設定した
