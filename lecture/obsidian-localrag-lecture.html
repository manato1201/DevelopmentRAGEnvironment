Problem

何が問題なのか

mcp-rag-serverはそのままでは「pgvectorに何が入っているか」を確認する手段がない。長期使用すると以下の問題が蓄積する。

問題影響現在の回避策
何がインデックスされているか不明古い情報・重複が気づかず蓄積なし
pgvectorの中身を確認できない検索精度劣化に気づけないなし
namespace別の状態が把握できないどこに何件あるか分からないなし
手動CLIでインデックス化が必要更新漏れが発生しやすい手動実行

解決方針

Obsidian vault = SOURCE_DIR とすることでファイルツリーがインデックス対象の一覧になる。watchdogがインデックス後にダッシュボードノートを自動更新することでpgvectorの状態をObsidianから常に確認できる。

Step 01

vault構造設計

namespaceとvaultフォルダを1対1で対応させる。管理専用の _rag_dashboard/ だけはインデックス対象から除外する。

obsidian-vault/
├── chat_logs/              ← namespace: chat_logs
├── tutorials/              ← namespace: tutorials
├── personal_notes/         ← namespace: personal_notes
│   ├── progress/
│   └── ideas/
├── private_docs/           ← namespace: private_docs
├── _rag_dashboard/         ← 管理専用（スキャン除外）
│   ├── index_status.md     ← 自動生成
│   ├── namespace_map.md    ← 自動生成
│   └── cleanup_log.md      ← 自動生成
└── _templates/             ← テンプレート（任意）
フォルダnamespace内容
chat_logs/chat_logsClaude等との会話ログ
tutorials/tutorialsHoudiniチュートリアル生成結果
personal_notes/personal_notesObsidianノート・個人進捗
private_docs/private_docs共有NG文書・草稿
_rag_dashboard/—管理専用（インデックス除外）
Step 02

SOURCE_DIR設定

.env の SOURCE_DIR をvaultルートに向けるだけ。これだけでObsidianのファイルツリーがRAGのソース一覧になる。

mcp-rag-server/.env
SOURCE_DIR=/home/tk_render/obsidian-vault
PROCESSED_DIR=/home/tk_render/obsidian-vault/_rag_dashboard/.processed

# 他の設定は変更不要
POSTGRES_HOST=localhost
EMBEDDING_MODEL=intfloat/multilingual-e5-large

PROCESSED_DIR を _rag_dashboard 配下に置く理由

処理済みファイルの記録もvault内で管理することで、Obsidianからインデックス処理の履歴を確認できる。

Step 03

frontmatterによる鮮度管理

各ノートの先頭にYAMLフロントマターを付けて、インデックス状態と有効期限を管理する。

tutorials/houdini_vex_wrangle.md
---
title: Houdini VEX wrangle 基礎メモ
namespace: tutorials
status: active
created: 2026-06-10
updated: 2026-06-10
expires: 2026-12-10
tags: [houdini, vex, tutorial]
rag_indexed: false
---

## 概要
...
フィールド値説明
statusactiveインデックス化する（デフォルト）
statusstale期限切れ・要確認。警告ログ付きでインデックス
statusarchivedインデックス化をスキップ
expires日付この日付を過ぎるとwatchdogがstaleに自動変更
rag_indexedfalse→trueインデックス完了後にwatchdogが書き換える

expiresの設定目安

チャット履歴: 90日 / チュートリアル: 180日 / 個人メモ: 無期限。古いチュートリアルが検索に混入するのを防ぐのが主目的。

Step 04

watchdog 改良版（書き戻し付き）

ファイルの変更を検知してインデックス化し、完了後にfrontmatterとダッシュボードを自動更新する。

auto_index.py — コアロジック
class VaultHandler(FileSystemEventHandler):
    def _should_skip(self, path):
        # _rag_dashboard / _templates / .obsidian はスキップ
        return any(s in path.parts for s in SKIP_DIRS)

    def on_modified(self, event):
        path = Path(event.src_path)
        if self._should_skip(path) or path.suffix != ".md":
            return
        fm = read_frontmatter(path)
        check_expires(path, fm)            # 期限チェック
        if fm.get("status") == "archived":
            return                         # アーカイブはスキップ
        if run_index():                    # インデックス化
            write_frontmatter_field(       # rag_indexed: true に書き戻し
                path, "rag_indexed", True
            )
            update_dashboard()             # ダッシュボード更新

インストールと起動

Ubuntu

Step 05

ダッシュボードノード仕様

watchdogがインデックス完了ごとに _rag_dashboard/ 内の3ファイルを自動更新する。Obsidianで開くだけで現在のRAG状態が分かる。

index_status.md
  • 最終更新日時
  • 総ファイル数
  • namespace別件数テーブル
  • staleファイルの一覧（Obsidianリンク付き）
namespace_map.md
  • namespace別のファイル一覧
  • インデックス済み: ✓ / 未: ○
  • status と tags を表示
  • Obsidianの[[リンク]]形式
cleanup_log.md
  • 削除・更新の履歴
  • タイムスタンプ付き
  • 追記式（消えない）

index_status.md の例

# RAG インデックス状態

総ファイル数: 47

Operations

運用フロー

日常的な使い方

1
Obsidianでノートを作成・編集

frontmatterに namespace と expires を記載して保存

2
watchdogが自動検知

保存と同時にmcp-rag-serverの差分インデックス化が走る

3
frontmatterとダッシュボードが自動更新

rag_indexed: true に書き換わり、index_status.md が最新になる

4
Claude Code から検索

mcp-rag-serverのMCPツールでnamespaceを指定して検索

月次メンテナンス

  1. _rag_dashboard/index_status.md を開いて stale ファイルを確認
  2. 各ファイルを開いて内容を精査（削除 or 更新）
  3. 更新した場合は updated と expires を書き換えて status: active に戻す
  4. 削除した場合は cleanup_log.md に自動記録される
