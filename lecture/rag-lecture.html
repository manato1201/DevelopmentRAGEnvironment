01 · Introduction

なぜ今、自前RAGを作るのか

AIツールが増えた結果、「AIに聞けば何でも答えてくれる」という幻想が生まれた。 しかし実務でぶつかる問題の大半は、汎用LLMが学習していない固有の情報に関係している。

🔴 LLMだけでは解決できないこと
  • 自分のチャット履歴・過去のアウトプット
  • 自分が作ったプロジェクトのドキュメント
  • 研究室固有のツール設定・規約
  • 学習カットオフ後の最新技術情報
🟢 RAGがあると解決できること
  • 「以前生成したチュートリアルと同じ形式で」
  • 「うちのHoudini環境に合わせたコードで」
  • 「先週の議事録を踏まえて提案して」
  • 「この論文の内容を参考に実装して」
実務家の視点

ゲーム会社に入ると、社内Wikiやプロジェクトドキュメントが膨大に存在する。 それを素早く参照して実装に活かせるエンジニアは、それだけで生産性が2〜3倍違う。 自分のRAG環境を作れるということは、どんな職場でも即戦力になれるということだ。

02 · What is RAG

RAGの仕組みを理解する

RAG = Retrieval-Augmented Generation

LLMに質問するとき、関連ドキュメントを検索して一緒に渡すことで、LLM固有の知識を超えた回答を得る技術。 


  ┌─────────────────────────────────────────────────────┐
  │                   RAG の流れ                        │
  └─────────────────────────────────────────────────────┘

  ① ユーザーが質問
       "HoudiniでVEXを使ったボロノイ分割の実装方法は？"

  ② 検索エンジンで関連ドキュメントを取得
       Vector DB / Notion API → 上位5件のドキュメント

  ③ プロンプトに context として注入
       "以下のドキュメントを参考に回答してください:\n
       [doc1]: Houdini VEX リファレンス...\n
       [doc2]: 過去のチュートリアル生成結果...\n
       \n質問: HoudiniでVEXを使ったボロノイ分割..."

  ④ LLM が context を参考に回答を生成
       → 自分の環境・過去の作業に即した具体的な回答

2種類の検索方式

方式 仕組み 精度 コスト 本システムでの使用
ベクトル検索 文章を数値ベクトルに変換し、意味的な近さで検索 高（意味的マッチ） Embedding生成が必要 ローカルRAG（pgvector）
キーワード検索 テキストの完全一致・部分一致で検索 中（表記ゆれに弱い） 低・実装シンプル クラウドRAG（Notion API）
よくある誤解

「ベクトル検索の方が絶対に良い」は正しくない。 Notion APIのキーワード検索でも、ドキュメントに適切なタグ・summaryが付いていれば十分実用的な精度が出る。 まず動くものを作って、精度が問題になってから改善するのが実務のセオリー。

参考実装: mcp-rag-server

今回のローカルRAGの基盤として使用する mcp-rag-server（松尾研究所）の特徴：

強み
  • LangChain非依存 → バージョン安定
  • multilingual-e5-large で日本語対応
  • 差分インデックス機能あり
  • CLI（管理）とMCP（検索）を分離
  • Dockerで環境を完全に封じ込め
改良点（本設計）
  • namespace追加 → 用途別に分離
  • watchdog → 自動インデックス化
  • 複数MCPホスト対応設定
  • Notion MCP との連携
03 · Design Philosophy

「何をどこに置くか」が全て

RAG設計で一番最初に決めるべきことは 「情報の分類」 だ。 ここを曖昧にすると、後から情報が混在して検索精度が落ち、最悪プライベートな情報が流出する。

分類基準: 共有できるか × 鮮度が重要か

情報の種類 共有 置き場所 理由
ツール仕様（Unity/Houdini/DX12） 共有すべき クラウド（Notion） チームで参照、更新頻度低
ゲーム情報・共有設計書 共有すべき クラウド（Notion） 複数人で使用する前提
手動精査した技術記事・論文 共有可能 クラウド（Notion） 品質担保済みのもの
チャット履歴・会話ログ 共有NG ローカル（pgvector） 試行錯誤・個人情報含む
チュートリアル生成結果 共有NG ローカル（pgvector） 個人の学習過程・未精査
Obsidianノート・個人進捗 共有NG ローカル（pgvector） 個人情報・草稿
設計原則

「迷ったらローカル」が正しい。クラウドに置いたデータは取り消せないが、 ローカルに置いたデータはいつでもクラウドに移動できる。 保守的に設計して、必要になってから公開範囲を広げるのが実務の鉄則。

04 · Cloud RAG

クラウドRAG: Notion + GAS + Gemini

なぜこの組み合わせか

Notion を選ぶ理由
  • 公式REST APIがある（GASから直接操作可能）
  • Notion MCPが公式対応済み
  • Web Clipperで手動追加が楽
  • database_id単位での検索で物理的に分離できる
GAS を選ぶ理由
  • 無料・サーバー管理不要
  • Googleアカウントだけで完結
  • WebApp・定期実行・フォームが全部できる
  • Notion API・Gemini APIの呼び出しも同一環境

DB設計: 4つのDBを完全分離

DB名内容追加方法
Tool Docs DBUnity・Houdini・DX12・環境設定手動 / スクリプト
Game Info DBゲーム仕様・共有ゲーム情報手動
Research DB論文・技術記事（手動精査済み）手動 / スクリプト
Team Notes DBゼミ資料・議事録・方針手動

各ページの共通メタデータ（重要）

後でGASから検索したとき精度が変わる。特に summary は収集時点でGeminiに生成させておくこと。

Notion ページプロパティ（全DB統一）
// 各DBページに必ず付けるプロパティ
{
  "title":        "ページタイトル",
  "source_url":   "https://...",            // 元URL
  "tags":          ["Unity", "Shader"],   // マルチセレクト
  "summary":       "AI生成の100字要約",      // 検索精度の核
  "collected_at":  "2026-06-10",
  "category":      "tool_docs"             // どのDBか
}
なぜ summary が重要か

Gemini APIへのリクエストはトークン数に比例してコストが上がる。 ページ本文をそのまま渡すと長すぎて精度も落ちる。 100字のsummaryを渡すことでコストを1/10以下に抑えながら精度を維持できる。

GAS RAGの実装フロー

1

ユーザーが質問 + 対象DBを選択

GAS WebAppのチャットUIで「Houdini関連を検索」など対象DBをドロップダウンで指定

2

Notion APIで指定DB内を検索

database_id を指定することでDB間の混合が物理的に防止される

3

上位N件のsummaryでcontextを構築

各ページのsummaryを連結してコンパクトなcontextを作る

4

Gemini APIに投げて回答取得

gemini-2.0-flash を使用。コスパ最良のモデル選択

GAS: RAGの核心部分
function ragQuery(query, dbKey) {
  // 1. 指定DBをNotionで検索
  const context = searchNotion(query, NOTION_DBS[dbKey]);
  
  // 2. プロンプト構築
  const prompt = `以下のドキュメントを参考に、日本語で回答してください。

【参考ドキュメント】
${context}

【質問】
${query}`;
  
  // 3. Gemini API 呼び出し
  const res = UrlFetchApp.fetch(
    `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.0-flash:generateContent?key=${GEMINI_KEY}`,
    {
      method: "post",
      contentType: "application/json",
      payload: JSON.stringify({
        contents: [{ parts: [{ text: prompt }] }]
      })
    }
  );
  
  return JSON.parse(res.getContentText())
    .candidates[0].content.parts[0].text;
}

ドキュメント追加の3パターン

🖱 手動（Notion UI / Web Clipper）

日常的な1件追加。ブラウザ拡張で閲覧ページを即Notionに保存。追加後にsummaryを手動or自動生成。

🐍 Pythonスクリプト → Notion API

まとめて追加・既存データ移行に使用。Research CollectorのRSS収集部分をそのまま流用し、出力先をNotebookLMからNotionに差し替えるだけ。

05 · Local RAG

ローカルRAG: mcp-rag-server 改良版

セットアップ手順（mcp-rag-server）

解説記事をベースに、 namespace設計と自動インデックス化を追加する。

1

リポジトリのクローン + uv環境セットアップ

Pythonのパッケージ管理はuvを使用（pip より高速・依存解決が安定）

2

Docker で pgvector を起動

Docker Desktop を使えばコマンド1行。Claude Code起動時に自動起動する設定も可能

3

.env の設定

DB接続情報・ソースディレクトリ・Embeddingモデルを設定。SOURCE_DIRをObsidian vaultに向ける

4

MCPホストに設定追加

Claude Code / Cursor / Cline それぞれの設定ファイルに同じエントリを追加。サーバー自体は1つを使い回す

Docker: pgvector 起動
# pgvectorコンテナを起動
docker run --name postgres-pgvector \
  -e POSTGRES_PASSWORD=password \
  -p 5432:5432 \
  -d pgvector/pgvector:pg17

# RAG用DBを作成
docker exec -it postgres-pgvector \
  psql -U postgres -c "CREATE DATABASE ragdb;"

namespace設計（用途別分離）

pgvectorのテーブルに namespace カラムを追加し、検索時にフィルタする。

namespace内容ソース
chat_logsClaude等との会話ログ自動エクスポート
tutorialsHoudiniチュートリアル生成結果生成時に自動保存
personal_notesObsidianノート・個人進捗Obsidian vault
private_docs共有NG文書・草稿手動追加
namespace指定での検索（SQL）
-- personal_notes だけを対象に意味検索
SELECT content, metadata
FROM documents
WHERE namespace = 'personal_notes'
ORDER BY embedding <-> query_embedding
LIMIT 5;

自動インデックス化（watchdog）

現状はファイルを追加するたびに手動でCLIを叩く必要がある。 watchdog ライブラリでディレクトリを監視し、変更を自動検知してインデックスを更新する。

auto_index.py（常駐プロセス）
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler
import subprocess, time

class IndexHandler(FileSystemEventHandler):
    def on_modified(self, event):
        # .md / .txt / .pdf の変更のみ対象
        if not event.is_directory and \
           event.src_path.endswith(('.md', '.txt', '.pdf')):
            print(f"変更検知: {event.src_path}")
            subprocess.run([
                "uv", "run", "python",
                "-m", "src.main", "index"
            ])

if __name__ == "__main__":
    observer = Observer()
    observer.schedule(
        IndexHandler(),
        path="./data/source",   # Obsidian vault を指定
        recursive=True
    )
    observer.start()
    print("監視開始...")
    try:
        while True:
            time.sleep(1)
    except KeyboardInterrupt:
        observer.stop()
    observer.join()

複数MCPホストへの接続

mcp-rag-serverは特定のLLMに依存しない設計のため、設定ファイルにエントリを追加するだけで どのMCPホストからも同じサーバーを使い回せる。

Cursor: ~/.cursor/mcp.json
{
  "mcpServers": {
    "mcp-rag-server": {
      "command": "uv",
      "args": [
        "run", "--directory",
        "/path/to/mcp-rag-server",
        "python", "-m", "src.main"
      ]
    }
  }
}
将来: Remote MCP対応後

claude.aiがRemote MCPに対応すれば、WebUI（claude.ai）からも同じローカルサーバーに接続できるようになる。 今の設計を維持していれば、その時点での変更はゼロ。

06 · Integration

クラウド↔ローカルの統合設計

参照方向のルール


  ローカル → クラウド（Notion）: ✅ 読み取り専用で参照OK
  クラウド → ローカル         : ❌ アクセス不可（意図的）

  理由: クラウドからローカルに書き込む仕組みを作ると、
        意図しないデータがローカルの個人情報と混在するリスクがある

Notion MCP による統合（今すぐ使える）

Notion MCPはclaude.aiに既に接続済みのため、 追加実装なしでClaude CodeからNotionのDBを直接検索できる。

実用例: Claude Codeでの使い方

「HoudiniのVEXで昨日作ったチュートリアルの内容（ローカル: tutorials namespace）と、 公式SOPリファレンス（Notion: Tool Docs DB）を両方参照して、ボロノイ分割の実装コードを書いて」

→ mcp-rag-server（ローカル）とNotion MCP（クラウド）を自動的に組み合わせて回答

将来拡張: Supabase Vector

チームへの展開が必要になった時点で検討する。今は不要。

フェーズ構成メリット
現在（個人） Docker + pgvector（ローカル） 無料・シンプル・管理コストゼロ
将来（チーム） Supabase Vector（クラウド） 複数人で同じVectorDBを参照・namespace分離維持
