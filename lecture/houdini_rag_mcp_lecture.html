Chapter 01

なぜ RAG が必要なのか

大規模言語モデル（LLM）は膨大なテキストで事前学習されていますが、知識の鮮度と専門ドメインの正確性という2つの根本的な問題を抱えています。

LLMの限界

問題 具体例（Houdiniの場合） 深刻度
知識のカットオフ Houdini 20.5 の新機能を知らない 高
ハルシネーション 存在しないノード名・パラメータを生成 高
バージョン混在 18.0〜21.0の情報が混在した回答 中
コンテキスト不足 独自HDAsの使い方を知らない 中
核心概念

RAG（Retrieval-Augmented Generation）は「検索して、それを根拠に生成する」アプローチです。LLMの推論能力はそのまま活かしつつ、グラウンディング（根拠付け）によってハルシネーションを抑制します。

ゲーム開発での実用価値

Houdini MCP でノードを自動配置する際のボトルネックは「存在しないノード・パラメータを指定してしまうこと」でした。ヘルプをRAGに読み込ませることで、LLMは正確なノード名とパラメータ名を参照しながらHoudiniを操作できるようになります。

Chapter 03

埋め込みモデルの仕組み

今回使用する intfloat/multilingual-e5-large は、テキストを 1024次元のベクトルに変換するモデルです。

意味的類似性とは

埋め込みモデルが実現するのは「意味が似ているテキストは、ベクトル空間上でも近くに配置される」という性質です。

直感的理解

「vellum solver のサブステップとは？」というクエリは、たとえ同じ単語が使われていなくても「Substeps: Number of solver substeps per frame.」というヘルプテキストと近いベクトル値になります。これがキーワード検索との根本的な違いです。

E5モデルの特徴

特徴 詳細
多言語対応 日本語・英語を含む100言語以上に対応
次元数 1024次元（large モデル）
Prefix設計 クエリには query: 、文書には passage: を付与する非対称設計

Prefix の重要性

Python — .env設定の意味 
# 検索クエリに付与するプレフィックス
EMBEDDING_PREFIX_QUERY="query: "

# インデックス対象ドキュメントに付与するプレフィックス
EMBEDDING_PREFIX_EMBEDDING="passage: "

# 実際の動作イメージ
query_input   = "query: vellum solver substeps"
passage_input = "passage: Substeps: Number of solver substeps per frame..."
注意

このプレフィックスを省略すると検索精度が大幅に低下します。E5モデルはこの非対称な入力形式を前提に学習されているためです。

Chapter 04

ベクトル検索アルゴリズム

コサイン類似度

2つのベクトル間の類似度を測る最も一般的な指標です。ベクトルの向き（方向）だけを比較し、大きさを無視します。

Math — コサイン類似度の計算 
# コサイン類似度の定義
cos(A, B) = (A · B) / (|A| × |B|)

# 値の範囲: -1.0 〜 +1.0
# +1.0 → 完全に同じ方向（最も類似）
#  0.0 → 直交（無関係）
# -1.0 → 真逆の方向

# Pythonでの実装例
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

HNSW インデックス（pgvectorの高速化）

単純な全件比較（ブルートフォース）では7810件×1024次元の計算が毎回必要になり実用的ではありません。pgvectorはHNSW（Hierarchical Navigable Small World）という近似最近傍探索アルゴリズムを使います。

HNSWの直感的理解

「六次の隔たり」理論に着想を得た構造です。「遠距離の高速道路ネットワーク」と「近距離の細道ネットワーク」を階層的に持ち、上位層から段階的に候補を絞り込むことで、全件探索の O(N) から O(log N) へ計算量を削減します。

手法 計算量 精度 用途
ブルートフォース O(N) 100% 小規模・評価用
HNSW（近似） O(log N) 〜99% 本番環境・高速検索
Chapter 05

pgvector と PostgreSQL

pgvector は PostgreSQL の拡張機能として動作し、通常のリレーショナルデータベースの機能にベクトル演算能力を追加します。

データモデル

SQL — ベクトルカラムの定義イメージ 
CREATE EXTENSION vector;

CREATE TABLE documents (
    id       SERIAL PRIMARY KEY,
    content  TEXT,                    -- 元テキスト
    source   TEXT,                    -- ファイルパス
    embedding vector(1024)          -- 1024次元ベクトル
);

-- コサイン類似度で近傍検索（上位5件）
SELECT content, 1 - (embedding <=> query_vec) AS similarity
FROM documents
ORDER BY embedding <=> query_vec
LIMIT 5;
演算子の意味

<=> はコサイン距離演算子です（類似度ではなく距離なので、値が小さいほど近い）。1 - (距離) で類似度スコアに変換できます。

Docker を使う理由

pgvectorのセットアップはOSへの直接インストールが複雑です。Dockerを使うことで pgvector/pgvector:pg17 イメージを1コマンドで起動でき、環境の再現性も保証されます。WSLの場合、DockerDesktopとのWSL Integration機能でWindowsのGUIからコンテナ管理ができます。

なぜMCPか — RAGとの関係

アプローチ 仕組み 課題
従来のRAG 検索→プロンプト構築→LLM呼び出しを1パイプラインで実装 LLMを変えるたびに実装し直し
MCP RAG 「ベクトル検索ツール」として公開。LLMが必要に応じて呼び出す ツール呼び出しの抽象化が必要
設計思想

MCPの重要な設計原則は「LLMはいつ検索すべきかを自分で判断する」という点です。RAGの検索タイミングをLLMの推論に委ねることで、「まず何が必要かを考え、必要なノードをリストアップしてからヘルプを検索する」という2段階推論が自然に実現されます。

WSL環境でのパス問題と解決策

Claude Desktop（Windows）からWSL内のサーバーを起動する際、bash -c で起動するとシェルの初期化が省略され uv コマンドが見つからない問題が発生しました。

JSON — 解決策：uvのフルパスを指定 
{
  "mcpServers": {
    "mcp-rag-server": {
      "command": "wsl",
      "args": [
        "bash", "-c",
        // PATHが通らないため ~/.local/bin/uv でフルパス指定
        "/home/tk_render/.local/bin/uv run --directory /home/tk_render/mcp-rag-server python -m src.main"
      ]
    }
  }
}
Chapter 07

実装手順の全体像

インフラ構築

  1. WSL2 + Ubuntu 24.04 を systemd 有効状態でセットアップ（/etc/wsl.conf に systemd=true）
  2. Docker Desktop をインストールし WSL Integration を有効化
  3. pgvector コンテナを起動し ragdb データベースを作成
  4. uv で mcp-rag-server を clone・依存関係インストール

データ準備

  1. Houdini インストールフォルダの houdini/help/ から zip ファイルを取得
  2. Python スクリプトで全 zip を展開
  3. .txt 以外のファイル（画像・動画等）を削除
  4. インデックス化実行（uv run python -m src.cli index）
時間の見積もり

7810ファイルのインデックス化は 4〜6時間 かかります。nohup でバックグラウンド実行し、tail -f index.log で進捗を確認しましょう。

使い方のコツ

問題 対策
日本語クエリで文字コードエラー 検索クエリを英語で指示する
抽象的タスクで検索失敗 「必要なノードを列挙してから各ヘルプを検索して」と段階を踏ませる
回答がタスク欄に出る 「Markdownファイルに出力して」と明示する

検索精度を上げるために

RAGの検索精度は以下の要素で決まります：

要素 チューニング方法 効果
チャンクサイズ --chunk-size を増やす 文脈が増えるが検索精度が下がりやすい
検索件数（Top-K） コード側で調整 多すぎるとLLMのコンテキストを圧迫
差分インデックス --incremental フラグ 追加ファイルだけを更新できる
